## § 0 TL;DR

1. **Production trace shape captured**: token_bucket_1k compiles to a single 8-block trace (`block0` entry / `block1` loop-head / `block2-block4-block6` hot-flow / `block3-block5-block7` deopt-leaves). 26 luna ops → ~80 cranelift instructions. Trace is bit-identical across 5 process reruns (only metatable hash addresses differ).
2. **Hot loop body has ZERO stores**. The back-edge chain `block1 → block2 → block4 → block6 → block1` emits **0** `store.i64` instructions. All 36 stores live exclusively in the 3 deopt-leaf blocks (each materialises all 12 slots before `return`).
3. **Hot loop body is helper-call bound**. Per iteration the trace makes 8 helper calls (5 table-index reads via `fn0/fn1/fn2/fn5/fn6` + 3 table-index writes via `fn3/fn4/fn7`) plus a handful of arithmetic + 3 brifs. **This is the actual perf gap**, not redundant slot stores.
4. **All 3 candidates from Phase 1G.C § 6.1 audit as NO-GO on this production shape**:
   - C1 (Loop-aware DSE): no stores in loop body → no surface to dead-store-eliminate.
   - C2 (brif-fragmented post-dom DSE): stores are in deopt-leaf blocks → must execute (materialise SSA → slots before side exit) → not dead-store-eliminable in any post-dom flavour.
   - C3 (Recorder-level register promotion): the recorder **already** does this — loop slots flow through `block1`'s 7 block-args + invariants stay in `block0`-loaded SSA Values. Nothing left to promote at the slot-write level.
5. **Phase 1I recommendation**: **abandon DSE-based attack surface entirely** on this benchmark. The methodology-correct next attack is **table-field IC (inline cache) on the helper calls** — what LuaJIT 2.1 does with `HREFK + HLOAD` after a metatable-stable guard. Effort: multi-week, multi-session. Phase 1I.A scope-down = "decompose LuaJIT's table-IC mechanism against luna's existing `Value::Table` + helper signatures". Path D's DSE thread is **complete-but-non-load-bearing** for token_bucket_1k.

### § 1.4 Why `find_dead_store_at` saw 0 invocations under Phase 1G.B

Phase 1G.B's cross-block branch fires when:
1. `state.get_last_store(func, current_inst)` finds a prior trusted store at the same MemoryLoc.
2. `prior_block ≠ current_block`.
3. `block_dominates(prior_block, current_block)` holds along the strict deopt-safe chain.

On this trace shape every store lives in a leaf block (`block3 / block5 / block7`). Leaves have **no successor blocks containing matching stores**. The (prior, current) pair where `prior_block` is one deopt leaf and `current_block` is a later store is unreachable — `prior_block` already executed `return` before `current_block` was even alive. Same-block path (Phase 1C) within a single deopt leaf has 12 stores at 12 distinct offsets (no duplicates), so it also has zero matches. The 0 fire count is the **structurally correct** outcome.

### § 1.5 Hot loop body cost decomposition (one iteration)

Counted from the IR (`block1 → block2 → block4 → block6` then back to `block1`):

| Cost | Count | Atomic-op-table µs (Apple M-series) | Sub-total |
|---|---|---|---|
| helper calls (table __index / __newindex) | 8 | ~100-200 ns each (variable; depends on metatable cache hit) | ~0.8-1.6 µs |
| `isub` / `imul` / `iadd` / `icmp` | 7 | ~1 ns each | ~7 ns |
| `iconst.i64` for stable addr literals | ~12 | folded into MOV-imm | ~5 ns |
| `brif` | 3 | ~1-3 ns (well-predicted) | ~6 ns |
| `jump` (back-edge) | 1 | ~1 ns | ~1 ns |
| stores | **0** in hot path | — | **0** |

Per-iteration wall-time on token_bucket_1k is dominated by the 8 helper calls — **~95% of the iteration cost** under the atomic-op-table model. The 1k-iteration outer loop pays this 1000 times = ~800-1600 µs total. This matches the measured runtime order-of-magnitude.

**Implication**: Any DSE-class optimisation (sub-2A, Phase 1C/1G.B/1G.C extension, or the 3 Phase 1H candidates) targets the **0%-store-cost surface** of the hot path. There is nothing to gain. The decomposition methodology guards against exactly this — "polish on the wrong attack surface".

## § 2 Phase 1H.2 — Candidate 1: Loop-aware DSE

### § 2.1 What this would do (per Phase 1G.C § 4.4 Option A)

Detect that consecutive stores to the same slot **across a back-edge** are dead-store-eliminable when (a) the slot is loop-private, (b) the back-edge is the only re-entry, (c) the kill happens before any read. Closer to LICM (Loop Invariant Code Motion) than to DSE per the Phase 1G.C verdict.

### § 2.2 Audit against measured shape

**Surface**: zero. There are **no stores in the loop body**. The back-edge `block6` is pure SSA `jump block1(args)` with no `store.i64`. Loop-aware DSE has nothing to eliminate because slot updates already flow through block-args.

### § 2.3 What Cranelift offers

Cranelift exposes `LoopAnalysis` (in `cranelift_codegen::loop_analysis`) and `DominatorTree` (in `cranelift_codegen::dominator_tree`). Both are available to `egraph.rs`. Building a loop-aware DSE pass on top is technically feasible (~2-4 weeks per Phase 1G.C § 6.1 estimate), but produces zero fires on this trace.

### § 2.4 Verdict

**NO-GO**. Effort: 2-4 weeks. Predicted production fires: 0. Gain: 0 µs. Family-1 risk (correctness): elevated (cross-iteration semantics + verifier interactions). **Cost/gain ratio: ∞**.
