1. **Production trace shape captured**: token_bucket_1k compiles to a single 8-block trace (`block0` entry / `block1` loop-head / `block2-block4-block6` hot-flow / `block3-block5-block7` deopt-leaves). 26 luna ops → ~80 cranelift instructions. 3. **Hot loop body is helper-call bound**. Per iteration the trace makes 8 helper calls (5 table-index reads via `fn0/fn1/fn2/fn5/fn6` + 3 table-index writes via `fn3/fn4/fn7`) plus a handful of arithmetic + 3 brifs. **This is the actual perf gap**, not redundant slot stores.
4. **All 3 candidates from Phase 1G.C § 6.1 audit as NO-GO on this production shape**:
   - C1 (Loop-aware DSE): no stores in loop body → no surface to dead-store-eliminate.
   - C2 (brif-fragmented post-dom DSE): stores are in deopt-leaf blocks → must execute (materialise SSA → slots before side exit) → not dead-store-eliminable in any post-dom flavour.

5. **Phase 1I recommendation**: **abandon DSE-based attack surface entirely** on this benchmark. The methodology-correct next attack is **table-field IC (inline cache) on the helper calls** — what LuaJIT 2.1 does with `HREFK + HLOAD` after a metatable-stable guard. Effort: multi-week, multi-session. Phase 1I.A scope-down = "decompose LuaJIT's table-IC mechanism against luna's existing `Value::Table` + helper signatures". | Block | Stores | Notes |

### § 1.4 Why `find_dead_store_at` saw 0 invocations under Phase 1G.B

Phase 1G.B's cross-block branch fires when:
1. `state.get_last_store(func, current_inst)` finds a prior trusted store at the same MemoryLoc.
2. `prior_block ≠ current_block`.
3. `block_dominates(prior_block, current_block)` holds along the strict deopt-safe chain.

On this trace shape every store lives in a leaf block (`block3 / block5 / block7`). Leaves have **no successor blocks containing matching stores**. The (prior, current) pair where `prior_block` is one deopt leaf and `current_block` is a later store is unreachable — `prior_block` already executed `return` before `current_block` was even alive. Same-block path (Phase 1C) within a single deopt leaf has 12 stores at 12 distinct offsets (no duplicates), so it also has zero matches. The 0 fire count is the **structurally correct** outcome.

### § 1.5 Hot loop body cost decomposition (one iteration)

Counted from the IR (`block1 → block2 → block4 → block6` then back to `block1`):

| Cost | Count | Atomic-op-table µs (Apple M-series) | Sub-total |
|---|---|---|---|
| helper calls (table __index / __newindex) | 8 | ~100-200 ns each (variable; depends on metatable cache hit) | ~0.8-1.6 µs |
| `isub` / `imul` / `iadd` / `icmp` | 7 | ~1 ns each | ~7 ns |
| `iconst.i64` for stable addr literals | ~12 | folded into MOV-imm | ~5 ns |
| `brif` | 3 | ~1-3 ns (well-predicted) | ~6 ns |
| `jump` (back-edge) | 1 | ~1 ns | ~1 ns |
| stores | **0** in hot path | — | **0** |

Per-iteration wall-time on token_bucket_1k is dominated by the 8 helper calls — **~95% of the iteration cost** under the atomic-op-table model. The 1k-iteration outer loop pays this 1000 times = ~800-1600 µs total. This matches the measured runtime order-of-magnitude.

**Implication**: Any DSE-class optimisation (sub-2A, Phase 1C/1G.B/1G.C extension, or the 3 Phase 1H candidates) targets the **0%-store-cost surface** of the hot path. There is nothing to gain. The decomposition methodology guards against exactly this — "polish on the wrong attack surface".

## § 2 Phase 1H.2 — Candidate 1: Loop-aware DSE

### § 2.1 What this would do (per Phase 1G.C § 4.4 Option A)

Detect that consecutive stores to the same slot **across a back-edge** are dead-store-eliminable when (a) the slot is loop-private, (b) the back-edge is the only re-entry, (c) the kill happens before any read. Closer to LICM (Loop Invariant Code Motion) than to DSE per the Phase 1G.C verdict.

### § 2.2 Audit against measured shape

**Surface**: zero. There are **no stores in the loop body**. The back-edge `block6` is pure SSA `jump block1(args)` with no `store.i64`. Loop-aware DSE has nothing to eliminate because slot updates already flow through block-args.

### § 2.3 What Cranelift offers

Cranelift exposes `LoopAnalysis` (in `cranelift_codegen::loop_analysis`) and `DominatorTree` (in `cranelift_codegen::dominator_tree`). Both are available to `egraph.rs`. Building a loop-aware DSE pass on top is technically feasible (~2-4 weeks per Phase 1G.C § 6.1 estimate), but produces zero fires on this trace.

### § 2.4 Verdict

**NO-GO**. Effort: 2-4 weeks. Predicted production fires: 0. Gain: 0 µs. Family-1 risk (correctness): elevated (cross-iteration semantics + verifier interactions). **Cost/gain ratio: ∞**.

## § 3 Phase 1H.3 — Candidate 2: brif-fragmented post-dom DSE

### § 3.1 What this would do (per Phase 1G.C § 4.4 Option B)

Relax the dominator-chain walk to a post-dominator walk on the store sink. Two stores `S1` (block A) and `S2` (block B) where A doesn't dominate B but every CFG path from A to S2 overwrites the slot before any read → DSE-able.

### § 3.2 Audit against measured shape

The 36 stores live in 3 mutually-exclusive deopt leaves (`block3 ⊥ block5 ⊥ block7`). Each leaf is a CFG sink — `return` immediately follows the 12 stores. Post-dominator analysis on these blocks:

- `block3` post-dominates nothing (it's a leaf with no successor).
- The hypothetical "Store in block X post-dominated by store in block Y" pattern needs Y to dominate the rest of the function after X. Here Y is a `return`-leaf, so the rest of the function after X is empty.
- Across deopt leaves the post-dom relationship is also empty (mutually exclusive paths from different brif guards).

**More importantly**: the stores in deopt leaves are **not dead**. They materialise the SSA-resident register state back to `reg_state` memory **before returning to the interpreter**. Removing them = leaving stale slot values = correctness bug. The interpreter reads `reg_state[A..A+B]` after side exit; if the trace didn't write `v18` back to `v0+8` before returning, the interpreter sees the old `bucket.last` and the program is wrong.

### § 3.3 Cranelift infrastructure for post-dom

Cranelift `DominatorTree` exposes only forward dominators. Post-dom is not built. Adding it = new dataflow pass (~1-2 weeks alone), then wiring into `alias_analysis.rs` (~1 week), then verifier integration audit (~1 week). Estimate: ~3-4 weeks total.

### § 3.4 Verdict

**NO-GO**. Effort: ~3-4 weeks. Predicted fires on this trace: 0 (no two stores form a post-dom pair where Y kills X's slot before a read). Even if fires occurred, removing deopt-leaf stores **breaks correctness**. Family-1 risk: SEVERE (deopt materialisation invariant violation). **Cost/gain: ∞ + correctness risk**.

## § 4 Phase 1H.4 — Candidate 3: Recorder-level register promotion

### § 4.1 What this would do (per Phase 1G.C § 4.4 Option C)

At luna's trace recorder level (`crates/luna-jit/src/jit_backend/trace.rs`), promote consecutive same-slot writes to a single `Variable` update + final write. Bypasses cranelift DSE by never emitting redundant stores in the first place.

### § 4.2 Audit against measured shape

The recorder **already does this**. Inspect block-args at `block1`:

```
block1(v18: i64, v25: i64, v26: i64, v27: i64, v29: i64, v30: i64, v31: i64)
```

Seven loop-carried slots flow as SSA block-args. The recorder uses `FunctionBuilder::def_var` + `use_var` for each register; cranelift's variable-to-SSA lowering then resolves them to block-args at merge points. The back-edge `jump block1(v56, v43, v60, v62, v60, v51, v36)` shows the updated values are passed directly — no stores in `block6`.

The "consecutive same-slot writes" pattern (e.g., load const → store slot → recompute → store slot again) **does not appear** in the loop body because the recorder uses Variables, not direct slot stores. The only stores in the trace are the deopt-leaf materialisation flush, emitted by the recorder once per side-exit block (`record_side_exit` path), not per source-op.

### § 4.3 Where stores still happen and why

Stores at deopt leaves are **necessary** (per § 3.2). They are the SSA→slot materialisation needed for the interpreter to resume correctly. The recorder emits them exactly once per side exit, not redundantly.

Sub-2A's failed dual-write (Phase 1F/1G.C) tried to ALSO write each `Op::Move` / `Op::LoadK` / `Op::LoadNil` to slot memory **inline** in the hot loop. That would have added stores **to the hot path that currently has none** — which is the opposite direction from "register promotion". Sub-2A was an anti-promotion. Phase 1G.B's DSE rule was designed to eat those redundant stores back. The fire-count gate caught the methodological error: there's nothing to eat because the surface is already promoted.

### § 4.4 What "further" register promotion could mean

The only remaining slot-write activity is the deopt-leaf materialisation. Hoisting that out is **impossible** — it's a correctness requirement of the side-exit contract. The recorder couldn't "promote" past this without redesigning the deopt ABI (multi-month, possibly intractable).

### § 4.5 Verdict

**NO-GO** — but for a positive reason: the work is already done. The recorder is at LuaJIT-2.1-equivalent for slot-write density on this trace shape. No further LOC change is meaningful here. **Cost: ~0, gain: ~0 — already at ceiling on THIS axis**.

### § 5.1 3-way matrix

The luna v2-J Chain A precedent (`~/.claude-shared/global/methodology/perf-decomposition-vs-polish.md` § 7 v2-J chapter) shows what happens when the methodology is ignored: 4 attacks predicted +12,500 µs, measured NEGATIVE on 4-5 cells, full revert.

### § 5.2 What the IR actually says the gap is

Per § 1.5 decomposition, ~95% of the hot loop's wall time is in **8 helper calls per iteration**. Those calls are:

- `fn0` = `__index(bucket, "last")` — read slot
- `fn1` = `__index(bucket, "rate")` — read invariant slot (could be hoisted if shape is stable!)
- `fn2` = `__index(bucket, "tokens")` — read slot (called 3× per iter: in refill branch + decrement check + decrement write)
- `fn3` = `__newindex(bucket, "tokens", ...)` — write slot
- `fn4` = `__newindex(bucket, "last", ...)` — write slot
- `fn5` = `__index(bucket, "tokens")` — read again
- `fn6` = `__index(bucket, "tokens")` — read again
- `fn7` = `__newindex(bucket, "tokens", ...)` — write again

This is the same gap LuaJIT 2.1 attacks with **HREFK** (hash-array reference, key-cached) + **HLOAD** + table-shape guards. After a metatable-stable guard, table field access lowers to a direct memory load at a known offset, not a helper call. Same pattern, ~10-50× cheaper per call.

### § 5.3 Recommendation: Phase 1I = table-field IC, not DSE

**Phase 1I direction**: **abandon Path D's DSE thread**. Open a new perf attack — **table-field inline cache** — modelled on LuaJIT 2.1's `lj_record_idx` + `IR_AREF`/`IR_HLOAD`/`IR_HSTORE` pipeline.

Concrete Phase 1I.A scope-down (single multi-session attack, audit-only first):

1. **Read** `lj_record_idx` + `lj_opt_fold` for table access patterns (~1 day, read-only, in `/private/tmp/luajit-src` if available, else WebFetch from LuaJIT github).
2. **Map** luna's `helper_index_get_*` / `helper_index_set_*` signatures (in `crates/luna-runtime-helpers/src/lib.rs`) against LuaJIT's `IR_HLOAD`/`IR_HSTORE` ABI.
3. **Audit** whether luna's existing `Value::Table` layout + metatable shape ID infrastructure (P16-F shape ID per CLAUDE.md) can support a stable-shape guard + offset cache.
4. **Decompose** token_bucket_1k's 8 helper calls under the atomic-op-table — what fraction is metatable-cache hit, what fraction is full lookup, what's the realistic per-call ceiling.
5. Produce Phase 1I.B decomposition doc with concrete attack list before any code change.

**No claim** here that table IC is a 1-week task. The Phase 1G/1H series budgeted ~6 weeks total and the answer is "wrong attack surface". Table IC is multi-month at minimum, with multiple sub-phases. The honest answer per the methodology is: this is the work, the prior 4 weeks were not wasted (they conclusively measured the DSE-axis ceiling), and the next attack starts with decomposition not code.

### § 5.4 Top-3 risks for Phase 1I

1. **Table IC has been attempted before in luna** (P16-F shape ID work per CLAUDE.md) — Phase 1I.A must audit what's already landed before re-implementing. Risk = duplicate work / accidentally regress an existing optimisation.
2. **luna's metatable model differs from LuaJIT's** — LuaJIT 2.1 has a single `GCtab` layout; luna has `LuaTable` + `LuaUserdata` + Phase TB trace-able userdata. The IC mechanism must handle both.
3. **Helper-call inlining at JIT trace level interacts with the inline-side-exit ABI** (Phase Stage 7 polish 6, per CLAUDE.md). If a helper call is inlined and the inline path takes a guard exit, the side-exit chain must materialise correctly. Family-1 deopt-correctness risk.
