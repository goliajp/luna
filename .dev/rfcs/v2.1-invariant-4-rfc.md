## Phase 2 â€” Cranelift 0.124.3 arm64 musttail audit

luna pins `cranelift = "0.124"` across `luna-jit`, `luna-aot`, `luna-runtime-helpers` (Cargo.tomls). Resolved version on disk: `cranelift-codegen-0.124.3`.

### IR-level support

`return_call_indirect` is defined as a first-class Cranelift IR opcode. aarch64 lowering rules (`src/isa/aarch64/lower.isle:2578-2601`):

```isle
;;;; Rules for `return_call` and `return_call_indirect`
(rule 1 (lower (return_call (func_ref_data sig_ref name (RelocDistance.Near)) args))
      (let ((abi Sig (abi_sig sig_ref))
            (uses CallArgList (gen_return_call_args abi args))
            (info BoxReturnCallInfo (gen_return_call_info abi name uses)))
        (side_effect (return_call_impl info))))

(rule (lower (return_call_indirect sig_ref ptr args))
      (let ((abi Sig (abi_sig sig_ref))
            (target Reg (put_in_reg ptr))
            (uses CallArgList (gen_return_call_args abi args))
            (info BoxReturnCallIndInfo (gen_return_call_ind_info abi target uses)))
        (side_effect (return_call_ind_impl info))))
```

### Emit sequence (aarch64)

`src/isa/aarch64/inst/emit.rs:3051-3065`:

```rust
&Inst::ReturnCallInd { ref info } => {
    emit_return_call_common_sequence(sink, emit_info, state, info);
    Inst::IndirectBr { rn: info.dest, targets: vec![] }.emit(sink, ...);
    sink.add_call_site();
    ...
}
```

`emit_return_call_common_sequence` (`emit.rs:3620-3666`) emits:

1. `AArch64MachineDeps::gen_clobber_restore(CallConv::Tail, ...)` â€” restore callee-saves per the Tail clobber map.
2. `Inst::LoadP64 { rt: fp, rt2: lr, mem: SPPostIndexed { simm7: setup_area_size } }` â€” `ldp fp, lr, [sp], #setup_area_size`.
3. Optional `gen_sp_reg_adjust(incoming_args_diff)` when tail arg slots differ.
4. Optional `auti_hint` (pointer-auth, if enabled).
5. `IndirectBr` â†’ `enc_br(rn)` = `BR Xn` (1 instruction).

For direct `Inst::ReturnCall` (`emit.rs:3036-3050`) the final instruction is `enc_jump26(0b000101, 0)` = unconditional 26-bit immediate `B target` plus an `Arm64Call` reloc â€” same as LuaJIT's `b`.

**Net per-tail-transition cost on M-series**: 1 `ldp` (~3-4 cycles) + 1 `BR` (1 cycle, perfectly predicted as RAS-disabled indirect-branch under known callsite distribution) + optional SP-adjust (1-2 cycles). Best-case ~5 cycles â‰ˆ 1.5 ns. Compare LuaJIT `b` â‰ˆ 1 cycle â‰ˆ 0.3 ns. Compare current luna `BLR Xn â†’ epilogue â†’ RET â†’ ORR â†’ RET` round-trip â‰ˆ ~92.92 ns/op per R3.3+ HALT measurement.

So Cranelift musttail does NOT match LuaJIT's bare-`b` per-transition cost exactly, but is **two orders of magnitude lower** than the current round-trip. â‰¥ 80% of the 89-ns gap is recoverable in theory.

### Verifier constraints

`src/verifier/mod.rs:1675-1700`:

```rust
fn typecheck_tail_call(...) -> VerifierStepResult {
    let signature = &self.func.dfg.signatures[sig_ref];
    let cc = signature.call_conv;
    if !cc.supports_tail_calls() {
        errors.report((... format!("calling convention `{cc}` does not support tail calls")));
    }
    if cc != self.func.signature.call_conv {
        errors.report((... "callee's calling convention must match caller"));
    }
    let types = signature.returns.iter().map(|param| param.value_type);
    self.typecheck_return_types(inst, types, errors, ...)?;
    Ok(())
}
```

Per `src/isa/call_conv.rs:76-82`:

```rust
pub fn supports_tail_calls(&self) -> bool {
    match self {
        CallConv::Tail => true,
        _ => false,
    }
}
```

**Only `CallConv::Tail` supports tail calls.** Both caller and callee SigRef must use it. Other implications:

- `CallConv::Tail` is "not ABI-stable except for exception payload registers" (`docs.wasmtime.dev/api/cranelift_codegen/isa/enum.CallConv.html`).
- wasmtime PR #8540 (merged 2024-05-03) flipped wasm engines to default-enable tail-call conv on x86_64/aarch64/riscv64.
- `supports_exceptions()` is true for both `Tail` and `SystemV`; exception payload regs on aarch64 = `x0, x1`.

### Toolchain ceiling â€” NOT blocking

Cranelift 0.124.3 + aarch64 + `CallConv::Tail` provides a stable, production-supported `return_call_indirect` lowering. The toolchain does not block Invariant 4.

### Where is trace-JIT-mode the production mode?

luna's default config sets `chunk_enabled = true` and `trace_enabled = true` (`crates/luna-core/src/vm/jit_state.rs:329-360`). Two production paths force `chunk_jit OFF / trace_jit ON` (= trace-JIT-mode = the mode where Invariant 4's gap lives):

#### Sandbox embedders (instr_budget enforcement)

## Phase 4 â€” Attack proposal (conditional) OR blocked verdict

### Verdict: PROCEED-CONDITIONAL

**Cranelift toolchain ceiling: NOT BLOCKED.** `return_call_indirect` on aarch64 with `CallConv::Tail` is a first-class, production-supported lowering since wasmtime PR #8540 (2024-05-03). luna's pinned 0.124.3 includes the full lowering, verifier check, and clobber-restore + branch emit.

### 3-phase implementation plan (gated on P1-redux PASS)

The plan below describes what an Invariant 4 attack would look like IF P1-redux surfaces a workload with dispatcher round-trip â‰¥ 5 Âµs/cell. Each phase is independently revertable; the whole chain is a separate sprint (â‰¥ 4-6 dev-weeks senior, per R3.3+ HALT charter).

#### Phase A â€” P1-redux decomposition (read-only, 5-10 dev-days)

- Pick one or two real workloads from the candidate list above (recommend: `sandbox + JSON parser` and `async_drive + producer/consumer message loop`).
- Build a per-workload `diag_*` example following `diag_fib28_wallclock_refresh.rs` shape.
- samply + bare-metal wall-clock + per-fn self-sample bucketing identical to Phase 10's recipe.
- Decomposition document (18+ stages, Â±20% budget audit) per perf-decomposition methodology Â§2.
- **Gate**: dispatcher round-trip â‰¥ 5 Âµs/cell named hot path. If not, declare Invariant 4 deferred and stop.

#### Phase B â€” Cranelift CallConv::Tail migration scaffold (15-25 dev-days)

If Phase A passes its gate:

- Add `CallConv::Tail` SigRef construction in `trace.rs` next to existing `module.make_signature()` calls (parent + child entry signatures).
- Define a new `TraceFn` ABI variant in `luna-core/src/jit/trace_types.rs`. Rust does not natively expose Cranelift's `CallConv::Tail`, so the entry from interp dispatcher must go through a small C-ABI trampoline that bridges `unsafe extern "C" fn(*mut i64) -> i64` â†’ Tail-conv body via Cranelift-emitted thunk. (Roughly one Cranelift-emitted entry block per trace + a fixed-size adapter at module init.)
- Restructure sentinel encoding: current parent does `bor (1<<63 | code<<56) into i64` post-`call_indirect`, then `return_`. With `return_call_indirect` the parent never sees the return, so sentinel must be encoded BEFORE the tail call. Options:
  - **(B-i) child encodes**: parent passes `sentinel_code` as a second arg; child ORs into its own return before `return_`. Requires every child trace's emit path to know about sentinel encoding. ~50 LOC across 19 callsites.
  - **(B-ii) TLS side-channel**: parent stores sentinel into a TLS slot before tail-calling; dispatcher reads from TLS instead of decoding return bits. Cleaner sentinel decoupling; ~20 LOC in dispatcher + 1 TLS write per tail callsite.
- Verifier-gated cargo test that `return_call_indirect` IR validates (test would emit a minimal Tail-conv parent/child pair and assert `Context::compile` succeeds).

#### Phase C â€” Hot-path migration + bench validation (10-15 dev-days)

If Phase B scaffolds clean:

- Migrate one callsite class at a time (TAG â†’ GLOBAL â†’ INLINE per v2-C-A7 gate classes) from `emit_side_trace_or_return`'s `call_indirect â†’ bor â†’ return_` pattern to `return_call_indirect`.
- Run the P1-redux workload bench after EACH callsite class migration (per perf-methodology Â§2 Phase B: bench validate after batches, not per-attack).
- Trip-wire: if any single batch shows â‰¥ 5 Âµs/cell regression at p50 over n=30 iter, revert that batch and stop.
- Acceptance: cumulative â‰¥ 30 Âµs/cell gain on the P1-redux workload AND no other workload (token_bucket, fib(28) default, redis_lua_shape) regresses > 2 Âµs/cell.

### Blocked-verdict fallback (if Phase A gates fail)

If Phase A finds dispatcher round-trip is NOT the named hot path on any real embedder workload (analogous to Phase 10's finding for token_bucket), declare Invariant 4 deferred to v2.2 with no LOC burnt. Handoff options:

- **Defer to v2.2 with explicit re-trigger**: only re-open if a future dogfood report names trace-JIT-mode dispatcher round-trip as a measured perf complaint.

- **Embedder-side mitigation**: document the trace-JIT-mode tradeoff in `docs/embedding.md` so embedders that hit this can opt into chunk_jit on (if they don't need instr_budget) or accept the gap.

## Top-3 risks if proceeding

1. **Repeat of v2-A â†’ v2-H polish disaster (perf-methodology Â§7).** Multi-week scaffold + migration on a workload that no real embedder actually runs would burn the same way the side-trace ABI sprint did. The mitigation is the Phase A gate (P1-redux must surface real-workload round-trip â‰¥ 5 Âµs/cell) BEFORE any LOC commit. Without that gate the attack is paper-confidence on fib(28)'s artificially heightened trace-JIT-mode gap. Mitigation strength: HIGH if the gate is enforced; LOW if Phase A is skipped under "we already know LuaJIT does it" pressure.

2. **CallConv::Tail is not C-ABI; the interpâ†’trace entry needs a trampoline.** Rust cannot natively express the Tail ABI, so the call from `vm/exec.rs` (`let entry_fn = ct.entry; ... entry_fn(reg_state.as_mut_ptr())` â‰ˆ line 6500) into a Tail-conv trace must go through a Cranelift-emitted adapter or a hand-written asm bridge. The adapter adds one call frame back at the interpâ†’trace boundary, partially canceling the savings for short-trace workloads. The win remains on **inter-trace** transitions, which is the actual Invariant 4 target. Risk: the adapter cost on the entry frame may exceed the savings on small workloads where most cells exit after 1-2 trace transitions. Phase A decomp must measure inter-trace transitions per cell, not total dispatch count.

3. **Sentinel encoding restructure interacts with deopt, exit_tags, per_exit_inline counting.** The current `bor (1<<63 | code<<56)` is a load-bearing protocol â€” `vm/exec.rs` decodes both sentinel and exit_tag from the i64 return; per-exit hit counters and inline-side-exit chains read from the same encoded value. Moving sentinel into a TLS side-channel (B-ii) or pushing it into the child (B-i) requires a touch across `exec.rs`, `jit_state.rs::JitCounters`, and the recording-frame deopt path. Any miss here = silent deopt misclassification = wrong inline-side-exit attribution = subtle perf regression that won't show on the P1-redux workload but will leak on others. Mitigation: Phase B trip-wire must include a deopt-correctness regression test (per-exit_tag counters before vs after must match for the same workload).

## What this RFC does NOT do

- **0 LOC luna code change**: this is research-grade scoping per task hard constraint. No `Cargo.toml` edit, no `trace.rs` edit, no test scaffold.
- **No gain numbers without measurement basis**: the "â‰¥ 80% of 89 ns recoverable in theory" is a Cranelift-emit-sequence estimate, not a measured gain. The actual workload-level Âµs/cell savings depends entirely on P1-redux's per-cell trace-transition count, which is unknown.
- **Does not commit luna to the attack**: the verdict is PROCEED-CONDITIONAL, gated on Phase A surfacing the workload. If Phase A finds nothing actionable, this RFC's job is to make the blocked-fallback path cheap (defer to v2.2 with no LOC burnt).
- **Does not merge with R3.3+ Design 1**: Invariant 4 is a different axis (dispatcher round-trip mechanism) than Invariant 1 (depth-baked SP refs). Design 1 retired per HALT verdict; Invariant 4 is a separate sprint.

## Methodology trigger-word audit (this RFC)

Per `~/.claude-shared/global/methodology/perf-decomposition-vs-polish.md` Â§1:

- **No "architectural ceiling" / "structural gap"**: the toolchain is named at instruction granularity (Cranelift 0.124.3 emit sequence, verifier rules, exact emit.rs lines).
- **No "language ceiling"**: Rust's lack of `extern "tail"` is named at mechanism level (the trampoline cost path), not as "Rust is slow".
- **No "user-space not touchable" / "kernel-bound"**: round-trip cost is fully attributed to user-space ABI traffic (BLR + epilogue + RET + ORR + RET).
- **No "sub-bench noise" / "within variance"**: gain estimate is per-instruction-cycle on M-series and explicitly named as "theoretical, pending P1-redux measurement".
- **No "deferred to deployer"**: deferral path is to v2.2 with explicit re-trigger conditions; embedder mitigation listed only as one option among three, not as primary escape.
- **No "workload not amenable" / "workload shape gates further perf"**: the question of which workload is amenable is the load-bearing question; this RFC's verdict is precisely "we don't know yet, find out via P1-redux".
- **No "self-baseline as progress yardstick"**: gain target is named against LuaJIT mcode `b` per-dispatch cost (~3 ns/op) and current luna trace-JIT-mode dispatcher cost (~92.92 ns/op per R3.3+ HALT measurement), absolute.
- **No "compile-time overhead bottleneck" hand-wave**: not invoked; Invariant 4 is a runtime-dispatch attack, not a compile-time attack.
- **No "perf metric semantics undefined"**: ns/op = ns per Lua bytecode op dispatched; Âµs/cell = Âµs per outer workload iteration. Consistent with Phase 10's semantics.
- **No "continuing v(N+1) polishing same subsystem"**: Invariant 4 is explicitly a DIFFERENT subsystem from R3.3+ Design 1 (dispatcher round-trip mechanism vs depth-relative SP refs). The HALT verdict named this exact distinction.
