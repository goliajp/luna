## TL;DR

luna's R3.3+ sub-2 (dual-write Variable) and sub-2B (`GlobalValue` + `MemFlags::trusted()`) attacks both landed NEGATIVE on a Cranelift 0.124 mid-end limitation: trivially-dead redundant stores survive the egraph store-DCE even with the strongest alias-disjointness hints the public API offers. This RFC audits FIVE alternative paths for the IF case where v2.2 needs a structural mid-end pass that Cranelift 0.124 cannot provide.

Headline matrix (effort in senior dev-weeks; risk H/M/L; perf potential is "what's recoverable on top of current sub-1 ship state"):

| Option | Effort | Risk | Perf potential | luna-core 0-dep compat | Closes Gap #1+#2 |
|---|---|---|---|---|---|
| **A. Cranelift upgrade** (0.125 → HEAD) | 0.5–2 dw | L–M | UNKNOWN — needs test build | YES (deps in luna-jit only) | UNKNOWN; public release notes signal NO change in trusted-store-DCE semantics |
| **B. LLVM JIT (inkwell)** | 12–26 dw | H | HIGH (proven SOTA store-DCE) | YES (luna-jit dep only); MASSIVE build-size + CI complexity cost | YES |
| **C. Hand-asm dispatcher (dynasm)** | 16–32 dw | H | HIGHEST per-transition; weakest IR optimization | YES | DIFFERENT — bypasses IR, no mid-end gap to "close" |
| **D. Cranelift custom mid-end pass** (ISLE fork) | 6–12 dw | H | TARGETED — close the specific gap | YES (vendored fork; supply-chain risk) | YES (Gap #1 cheap; Gap #2 harder but enumerable) |
| **E. WASM via cranelift-wasm** | 8–16 dw | H | LOW — adds layer, doesn't fix gap | YES | NO — same backend underneath |

**Recommendation for v2.2 (conditional on a structural close-the-gap being a hard requirement)**: **A first** (cheap probe, ≤1 dw). If A NEGATIVE → **D second** (custom pass, addresses root cause). Decline B (cost-blocking, multi-month + 10–20× binary-size growth), C (orthogonal — different attack surface; only relevant if Invariant-4-style tail-call attack), E (does not address the gap, only adds a layer).

**Honest fallback if A + D both NEGATIVE**: do NOT switch backends. Switch attack layer instead — Design 2/3 re-decomposition per `v2.0-track-r-r3-3-design-space.md` at a layer that doesn't depend on mid-end store-DCE (e.g., emit base shift as per-callsite immediate the dispatcher fixes up at runtime, not via a `base_var` indirection).

## §1 Phase 1 — Evaluation criteria extracted from R3.3+ verdicts

This section extracts the SPECIFIC Cranelift 0.124 gaps luna hit, so alternatives can be evaluated against them by mechanism, not by vibes.

### §1.1 Gap #1 — Mid-end does not DCE redundant trusted stores to the SAME address

Source: `v2.0-track-r-r3-3-sub2B-verdict.md` §3.2, Phase F IR (`block1`, `move_then_mul_propagates_via_move` fixture):

```
v8 = load.i64 notrap aligned v7+24     ; Op::Move LOAD
store notrap aligned v8, v7            ; *** sub-2B Op::Move STORE  (v7+0)
v11 = imul.i64 v9, v10                 ; Op::Mul reads Variable shadow
store v11, v0                          ; *** Op::Mul store-back     (v0+0)
store.i64 v10, v0+8                    ; deopt store-back
store.i64 v12, v0+16                   ; deopt store-back
store.i64 v13, v0+24                   ; deopt store-back
store.i64 v14, v0+32                   ; deopt store-back
v15 = iconst.i64 0
return v15
```

`v7 -> v0`, so `v7+0 == v0+0`. Two consecutive stores to the **same** address, no intervening load of that address, and Cranelift 0.124 mid-end **keeps both**. The first store is trivially dead — overwritten by the second with no observable side-effect window between them.

`MemFlags::trusted()` was the strongest disjointness hint the public 0.124 API permits short of declaring an alias region: it sets `notrap + readonly` flags and clears the alias region. It still does not unlock store-DCE.

**Per-iter cost attribution** (sub-2A verdict §4.1): 4–5 surviving stores per migrated arm × 1000-iter token_bucket hot loop ≈ 4500 redundant `str Xd, [Xn, #imm]` × ~1.5 ns each = ~6–7 µs/run. Matches measured +4.75 µs drift-adjusted regress within atomic-op-cost-table budget.

**Evaluation criterion for alternatives**: does the candidate backend's mid-end DCE redundant trusted stores to the same address when no intervening load aliases the address?

### §1.2 Gap #2 — `GlobalValueData` cannot wrap a runtime function param

Source: `v2.0-track-r-r3-3-sub2B-verdict.md` §3.1.

`GlobalValueData` (`cranelift-codegen 0.124.3/src/ir/globalvalue.rs:14`) has five variants: `VMContext`, `Load`, `IAddImm`, `Symbol`, `DynScaleTargetConst`. All five represent compile-time-resolvable addresses; none wraps a runtime SSA Value (e.g. luna's `reg_state` block param).

The only workaround = switch the trace signature param to `ArgumentPurpose::VMContext` and declare a `VMContext` GV. That ABI shift cascades through:

- trace signature (`trace.rs:5078-5082`)
- `call_indirect` sig_ref for side-trace tail-calls (`trace.rs:5134-5139`)
- dispatcher in `luna-core/src/vm/exec.rs` (treats param as plain I64)
- AOT export linkage path (expects normal I64)
- `JITModule:Send` Phase 6 prep interaction

Multi-day infra, far beyond sub-2B scope and entangling Phase 6 work.

**Evaluation criterion for alternatives**: does the candidate backend let us hand the optimizer a "this pointer is a base, treat as alias-disjoint" hint WITHOUT an ABI-purpose marker that cascades through every callsite?

### §1.3 Gap #3 — Dual-write Variable shadow correctness window forces semantic overhead

Source: `v2.0-track-r-r3-3-sub2-verdict.md` §2.2 + §4.1; `v2.0-track-r-r3-3-sub2B-verdict.md` §3.3.

luna's gradual migration uses Variable shadows + memory dual-writes per arm during incremental op-arm migration. The Variable shadow is needed because un-migrated op-arms still read from the Variable, not from memory; a pure-memory migration breaks correctness (sub-2B §3.3: `move_then_mul` returns `5994` = `999*6` instead of `7*6=42`, because Op::Mul reads slot 0 via Variable shadow = stale prelude-load value while Op::Move only wrote to memory).

The dual-write cost itself is small per-store (~1.5 ns Apple Silicon `str Xd, [Xn, #imm]`), but accumulates: 4–5 surviving stores per migrated arm × 1000-iter loop ≈ ~7 µs/run.

**Evaluation criterion for alternatives**: does the candidate backend either (a) allow the migration to be one-shot (no incremental gradual-migration window, e.g. by emitting from a higher-level IR that's not op-arm-incremental) or (b) have a stronger store-DCE that DCEs the dual-write in cases where the optimizer can prove redundancy?

### §1.4 Gap #4 — `return_call_indirect` musttail support exists at toolchain level but workload prereq unproven

Source: `v2.1-invariant-4-rfc.md` Phase 2 + Phase 3.

Cranelift 0.124.3 lowering for `return_call_indirect` on aarch64 with `CallConv::Tail` is production-supported (wasmtime PR #8540, 2024-05-03). The emit sequence is `clobber-restore + ldp fp,lr + maybe SP-adjust + BR Xn` — functionally tail, ~5 cycles vs LuaJIT bare-`b` 1 cycle. Two orders of magnitude lower than current luna `BLR → epilogue → RET → ORR → RET` ≈ 92.92 ns/op round-trip per R3.3+ HALT measurement.

Toolchain ceiling is NOT a blocker for Invariant 4. The blocker is workload prerequisite (P1-redux must surface dispatcher round-trip ≥ 5 µs/cell on real sandbox/async embedder).

**Evaluation criterion for alternatives**: Invariant 4 is workload-gated, not Cranelift-gated. The backend question is orthogonal here. Included in the matrix only so any alternative's tail-call story is documented; Invariant 4 is NOT a justification to switch backends.

### §1.5 Summary — the "do we need a different backend?" question

The honest reading: **only Gap #1 and Gap #2 are Cranelift-specific blockers**. Gap #3 is partially solved by a stronger Gap #1 fix (better store-DCE makes dual-write affordable). Gap #4 is workload-gated.

If an alternative solves Gap #1 AND Gap #2 at acceptable cost, it unlocks R3.3+ sub-2 path. If it doesn't, switching backends is pure cost. luna-core has a 0-third-party-dep contract; the JIT-related crates can take deps but build dep size still matters (Track L lint cleanup + binary size are explicit project constraints).

## §2 Phase 2 — Cranelift upgrade path audit (0.125 → HEAD)

### §2.1 Released versions since 0.124

Cranelift releases ride wasmtime's roughly-monthly cadence. As of the audit date, recent versions are:

- **0.124.x** — pinned by luna (`luna-jit`, `luna-aot`, `luna-runtime-helpers`); resolved on disk `0.124.3`.
- **0.125.x** — minor release, ships with wasmtime 38.x.
- **0.126.x** — minor release, ships with wasmtime ~39.x.
- **HEAD on `main`** — bytecodealliance/wasmtime main, version field rolls toward next minor.

### §2.2 Relevant areas of the mid-end (egraph / alias-analysis / store-DCE)

Cranelift's mid-end has had active alias-analysis and egraph rework in the 0.123 → HEAD window. Areas plausibly relevant to Gap #1 / Gap #2:

- **alias-analysis pass** (`cranelift-codegen/src/alias_analysis.rs`): handles load-forwarding and store-elimination within an extended basic block. The 0.124 design is "best-effort intra-EBB, no inter-block alias reasoning". Gap #1's IR (`block1` only) IS intra-EBB, so alias-analysis is in-scope for this specific case.
- **egraph store rewrites** (`cranelift-codegen/src/egraph/`): rewrites stores as part of GVN/equality saturation. The 0.124 egraph is store-conservative: it does not assume two stores to the same SSA-equal pointer with the same `MemFlags::trusted` are redundant. Wasmtime issue tracker has periodic discussion about loosening this; the public rationale is that "trusted alone does not prove the stored region isn't read by an external observer (e.g. signal handler) between the two stores".
- **`MemFlags` extensions**: the API has grown `alias_region` (`Heap` / `Table` / `VMContext`) to give the optimizer explicit alias-disjointness regions. Whether HEAD adds new region tags or DCE rules is the load-bearing question for upgrade.

### §2.3 Wasmtime issue tracker — keyword scan and public-release-note read

Phase 2 of this audit performed targeted WebSearch + WebFetch on the following queries / pages to confirm or refute whether 0.125–HEAD closes Gap #1:

- "wasmtime cranelift trusted store DCE"
- "wasmtime cranelift alias analysis mid-end"

- "wasmtime cranelift MemFlags region"

- bytecodealliance/wasmtime `RELEASES.md` recent entries.

Specific public-source findings:

- **bytecodealliance/wasmtime issue #4166** (May 2022, "Cranelift: alias analysis: track each individual table/heap separately") — proposes finer-grained alias regions in MemFlags. No merged PR identified in the 2024–2026 window via public web search. Status: open / progress unclear from public surface.
- **bytecodealliance/wasmtime issue #6661** (open, "Cranelift: DCE unused stack slots") — explicitly tracks STACK-SLOT DCE, NOT store-DCE. This is the closest tracked work to Gap #1, but addresses a different concern (eliminating the slot itself when unused). The status (open) signals the broader DCE story remains a work item.
- **bytecodealliance/wasmtime issue #4128** ("Cranelift: develop a mid-end optimization framework with passes in ISLE") — landed as the foundational mid-end / egraph work referenced by the April 2026 cfallin blog post.
- **cfallin April 2026 blog post** "The acyclic e-graph: Cranelift's mid-end optimizer" — explicitly documents the egraph's rewrite passes including "redundant load elimination" and "store-to-load forwarding", but does NOT discuss dead-store elimination of two consecutive same-address stores. The post's framing is sea-of-nodes + union nodes + extraction — the rewrite library is designed around value-flow not around store-DCE. **This is direct evidence that Gap #1 is not in the egraph's current published feature set.**
- **cranelift-codegen 0.125.3 docs.rs page** does not surface a per-version changelog. Detailed change list would require commit-log diff between 0.124.3 and 0.125.x — out of scope for read-only audit without a test build.

Synthesized findings:

- **0.125 / 0.126 minor release headlines** (per public release-notes scan): dominated by wasm-spec compliance, GC heap reform, component-model wiring, exception-handling progress. No headline entry "trusted store DCE" or "redundant trusted store elimination".
- **alias-analysis pass on `main`**: continues the intra-EBB best-effort design first landed in 2022 with the four-bit MemFlags region categorization (heap / table / vmctx / other). No public commit-message confirms an upgrade to whole-function alias reasoning between 0.124 and the audit date.
- **MemFlags additions in 0.124 → HEAD**: documented additions focus on alias regions for wasm GC types (struct/array fields) and exception-handling payload regs, not on a new "alias-disjoint scratch buffer" region applicable to luna's `reg_state`-relative loads/stores.
- **egraph store-elimination on `main`**: rewrites handle load-forward (load after store of known value) and constant folding; explicit dead-store elimination across two consecutive same-address stores remains absent from the published feature set per the April 2026 cfallin blog post.

**Confidence**: medium. Best public-source evidence available without a test build. To upgrade to high confidence, a fork-build probe is needed (option A's recommended ≤1 dw spend).

Bump `cranelift-codegen` / `cranelift-frontend` / `cranelift-module` / `cranelift-jit` / `cranelift-native` versions across luna-jit, luna-aot, luna-runtime-helpers Cargo.tomls. ## §3 Phase 3 — LLVM JIT audit (inkwell / llvm-sys)

### §3.1 Integration overview — what does a trace JIT migration entail?

luna's current trace JIT structure (per `crates/luna-jit/src/jit_backend/trace.rs`):

1. **Recording phase**: bytecode interpreter writes a linear op-list per trace into `RecordingTrace`. Live-out variables tracked via `var_table`; per-op kinds via `current_kinds[]`.
2. **Lowering phase**: `lower_trace_into_named` walks the op-list and emits Cranelift IR via `FunctionBuilder` + `cranelift_frontend::Variable` shadows. Each op-arm has a Cranelift-builder snippet.
3. **Compilation phase**: `JITModule` finalizes the function, returns `*const u8` code pointer; cast to `TraceFn = unsafe extern "C" fn(*mut i64) -> i64`.
4. **Dispatch phase**: `vm/exec.rs` calls `entry_fn(reg_state.as_mut_ptr())` on trace entry; sentinel return code drives next-trace selection.

An LLVM JIT migration would replace items 2–3 wholesale. inkwell wraps LLVM's C API into Rust types; `Builder<'ctx>` replaces `FunctionBuilder`; LLVM SSA `Value`s replace Cranelift `Value`s; LLVM IR `store` / `load` replace Cranelift's; ORC v2 / LLJIT replace `JITModule`.

Key differences relevant to Gap #1 / Gap #2:

- **LLVM `tbaa` (type-based alias analysis) metadata**: `!tbaa !N` on a load/store names an alias group; the optimizer is then free to assume disjoint groups don't alias. This maps directly to luna's `reg_state` as a stack-disjoint root for register slots.
- **LLVM `noalias` attribute** on pointer arguments propagates through SSA; combined with `!alias.scope` and `!noalias` metadata on loads/stores, LLVM provides precisely the "this pointer is a base, treat as alias-disjoint" hint Gap #2 needs WITHOUT an ABI-purpose marker.
- **LLVM DSE (dead-store elimination) pass** is a long-established optimizer pass (`DeadStoreElimination.cpp`) that handles same-address back-to-back stores trivially when alias info confirms disjointness with intervening loads/calls.

So LLVM almost certainly closes Gap #1 AND Gap #2 per published optimizer behaviour (not measured on luna).

### §3.2 Build dep size implications

LLVM is huge. Quantitative:

- **llvm-sys static**: links against system LLVM (`llvm-config --link-static` adds ~25–80 MB to the binary depending on which components are linked). Requires LLVM installed on dev machine + CI matrix.
- **llvm-sys dynamic**: links `libLLVM-NN.so/dylib` (~150 MB shared lib on disk). Binary itself stays slim but runtime requires libLLVM matching the major version.
- **inkwell**: pure Rust safe wrapper over llvm-sys; adds ~30 K LOC of safe wrappers; `cargo build` slowdown ~2× over current 0.124 cranelift dep tree.
- **LLVM version pin**: inkwell requires a specific LLVM major (e.g. `llvm17-0` feature ↔ LLVM 17). Bumping LLVM requires bumping inkwell minor + re-validation. CI matrix grows.
- **AOT path concern**: luna-aot currently produces small `.a` static libs for embed (Alpine staticlib is 1.4 MB). Linking against LLVM blows that up unless LLVM is treated as a JIT-only dep — but then luna-aot has to drop LLVM entirely and stay on Cranelift, which is a worse architectural split than the current state.

luna-core 0-dep is preserved (LLVM deps would live in luna-jit only). But luna's published binary size is currently a feature (12 MB Windows PE, ~5 MB Linux); LLVM-as-JIT-backend ≈ 10–20× binary size growth even with dynamic linking via `libLLVM-NN.dylib` because dev/CI must ship the runtime.

### §3.3 License implications

- **LLVM project license**: Apache-2.0 with LLVM exception. Apache-2.0 is permissive; LLVM exception relaxes Apache's anti-patent-retaliation clause to address a known concern about Apache vs GPLv2 compatibility. Compatible with luna's MIT-or-Apache-2.0 dual-license.
- **llvm-sys**: MIT.
- **inkwell**: Apache-2.0.

License-wise, no blockers. `cargo deny` config would need to add `Apache-2.0 WITH LLVM-exception` to the allowed license list (or use the SPDX `expr` matcher).

### §3.4 Memory model + thread safety vs current Cranelift

- **Cranelift current state**: `JITModule:Send` Phase 6 prep verdict notes Option B viable (multi-month). luna's trace JIT presently relies on single-threaded `JITModule`; Send-ification is gated work.
- **LLVM ORC v2 / LLJIT**: `LLJIT` and the underlying `ExecutionSession` are thread-safe by design (ORC = "On-Request Compilation"; concurrency is a first-class goal). Compiling a trace from a different thread is supported via `ThreadSafeModule`. Materially better than Cranelift's current state for luna's eventual multi-thread VM work.
- **Caveat**: LLVM's mid-end optimizer passes are NOT all thread-safe; concurrent compilation requires per-thread `LLVMContextRef` or use of `ThreadSafeContext`. Migration complexity rises with multi-thread ambition.

### §3.6 Drawbacks (honest)

- **Binary size**: 10–20× growth even dynamic-linked. luna's "12 MB Windows PE" / "1.4 MB Alpine staticlib" features become "150+ MB libLLVM required".
- **CI matrix**: every supported target (Apple Silicon, Linux x86_64, Linux aarch64, Alpine musl, Windows MinGW, Windows MSVC, WASI) must have an LLVM toolchain installed; current CI bundles Cranelift via cargo. CI complexity grows substantially.
- **Compile time per trace**: LLVM's full optimizer is 10–100× slower than Cranelift per IR-byte. For a JIT, this matters: a trace that took 200 µs to compile on Cranelift may take 5–10 ms on LLVM-O2. For token_bucket-shape traces (~30 ops, compile-rare) this is fine; for fib(28)-shape (many small traces), this is a noticeable latency hit. Mitigation = LLVM `-O1` or `-O0`, but then store-DCE quality drops back near Cranelift level.
- **AOT split risk**: if luna-aot stays on Cranelift but luna-jit moves to LLVM, the two backends drift in codegen shape (different load/store patterns, different calling conventions). AOT-vs-JIT correctness cross-validation becomes harder.
- **Track L 0-dep gate** doesn't break (LLVM deps in luna-jit), but inkwell pulls a transitive dep tree that includes `libc`, `libloading`, etc. — manageable but not free.
- **Multi-month timeline**: 12–26 dw ≈ 3–6 senior dev months. Largest cost option in the matrix.
- **Reversal cost**: if LLVM migration goes negative (perf gain doesn't materialize at workload level), reverting is multi-week.

YES. LLVM deps live in luna-jit only; luna-core stays 0-dep.

## §4 Phase 4 — Hand-asm + Cranelift custom passes + WASM

### §4.1 Hand-asm dispatcher (dynasm reference: LuaJIT)

LuaJIT 2.1's interp dispatcher is hand-written assembly per-arch (`src/vm_arm64.dasc`, `src/vm_x64.dasc`, etc.) using DynAsm (DSL preprocessor that emits .h headers from `.dasc` source). Each ISA has one `.dasc` file roughly 4–6 K LOC.

For a luna trace JIT migration:

- DynAsm-rs crate exists (Rust port of LuaJIT's DynAsm); generates inline-asm via macros at Rust build time.
- Alternative: emit machine code byte-by-byte via custom `AssemblerBuf` (more work per arch).
- Per-arch effort: roughly 4–8 dw per ISA for the trace JIT emit paths (Move, LoadK, LoadNil, Add, Sub, Mul, Lt, Le, Eq, Call, Return, Idx, GetField, SetField, Concat ≈ 15 ops × 1–2 days per ISA + dispatcher + side-exit chain).
- Supported arches we need: arm64 (Apple Silicon, AOT, primary), x86_64 (Linux/Windows/CI), RISC-V (optional, defer). That's 2 ISAs × 4–8 dw = 8–16 dw just for the ISA-specific paths.
- AOT path: hand-asm-only means luna-aot must duplicate the asm or emit via a separate codegen path. For AOT, either LLVM-static-bin path or Cranelift-as-AOT-only remains.

**Closes Gap #1?** Different attack surface. Hand-asm bypasses IR entirely: there is no mid-end DCE pass at all because there is no IR. The dual-write window of Gap #3 simply doesn't apply because per-arm emit is direct asm that writes only the needed registers/memory. You can hand-emit the desired pattern (one store per op-arm to slot) directly — no Cranelift mid-end DCE needed because there's no redundant store to DCE.

So hand-asm SOLVES Gap #3 (no dual-write) and trivially doesn't have Gap #1/#2 (no IR to DCE / no GlobalValue). But it gives up:

- Cross-arch portability via IR.
- Future-proof codegen on new arches (every new arch needs its own .dasc).
- Whole-trace optimizations (constant folding, GVN, register allocation across ops) — you only get what you hand-roll.

### §4.2 Cranelift custom mid-end pass extension API

Question: does cranelift-codegen expose a public extension API for registering custom mid-end passes?

Audit (read-only, based on cranelift-codegen 0.124.3 source structure):

- **Public API**: `Context::compile` and `Context::optimize` are the entry points. `optimize` runs a hard-coded pass pipeline (legalize → alias-analysis → egraph → DCE → regalloc). There is no `register_custom_pass()` or `add_pass(Box<dyn Pass>)` in the public API.
- **Internal pass plugin point**: passes are statically linked in the optimize pipeline; adding a custom pass requires forking cranelift-codegen and adding the pass in the pipeline source.
- **Egraph rewrite rules**: the egraph has a rule DSL (`cranelift-codegen/src/egraph/elaborate.rs` + `src/opts/*.isle`). Custom rules can be added by patching `opts/*.isle` and rebuilding cranelift. This is the LEAST-invasive customization path — single-file changes to `.isle`, recompile cranelift-codegen, link.

So the answer to "does the extension API exist?" is **NO public API**, but the egraph ISLE files are extensible in a fork.

Effort estimate for a custom mid-end pass closing Gap #1:

- 0.5 dw — write the ISLE rule for "store X to addr A; store Y to addr A with no intervening load → DCE first store" (intra-EBB only; respects trusted-flag if needed).
- 1–2 dw — set up a vendored cranelift-codegen fork (git submodule or workspace patch); CI must build from the fork.
- 2 dw — test the rule against luna's sub-2B Phase F IR fixture; verify no regression on existing trace fixtures (token_bucket, fib(28), 9 RFC cells).
- 1–2 dw — upstream the rule to bytecodealliance/wasmtime as a PR (so we eventually drop the fork). Upstream review cycle: 1–3 months wall-time, but luna ships on the fork in the meantime.
- 2–4 dw — additional rules for Gap #2 (better alias-region for stack-disjoint scratch buffer like `reg_state`). This may require new MemFlags region tags + alias-analysis pass updates — much harder than the single ISLE rewrite.
- 1 dw — supply-chain audit, cargo-deny config update for the vendored fork.

**Total: 6–12 dw**, dominated by the Gap #2 mid-end work (Gap #1's intra-EBB DCE rule is much cheaper at ~3 dw).

### §4.3 WASM via cranelift-wasm

Concept: emit luna trace IR as WASM bytecode, then pipe through `cranelift-wasm` → cranelift IR → JITModule. The hope is that wasm-spec semantics force a more aggressive optimizer pipeline.

- **Closes Gap #1?** No. cranelift-wasm lowers WASM → Cranelift IR; the same 0.124 mid-end runs underneath. Phase F IR audit would show the same surviving stores.
- **Adds layer**: WASM-byte emit + parse + lower adds a per-trace step costing ~50–200 µs at compile time. Runtime emitted code is no better than direct Cranelift IR emit.
- **Possible upside**: wasm's stricter alias model (linear memory disjoint from local SSA) MIGHT exercise stronger alias-analysis paths in Cranelift than the raw IR path. Speculative; would need test.
- **Drawback**: WASM's value types (i32/i64/f32/f64 only) don't natively express luna's Lua values (tagged i64 / boxed pointers). Encoding overhead per op.

**Effort: 8–16 dw.** Mostly the new emit path + parse glue + value-type encoding.

**Verdict**: cute, but does not close the mid-end gap underneath. Skip unless WASM is independently desired (e.g. for WASI-target embed).

### §4.4 luna-core 0-dep compat (all three §4 options)

- Hand-asm via dynasm-rs: dynasm-rs is a proc-macro at build time, no runtime dep on luna-core.
- Cranelift custom passes: same as upstream Cranelift, deps in luna-jit only.
- WASM via cranelift-wasm: adds `cranelift-wasm` to luna-jit Cargo.toml only.

All three preserve luna-core's 0-dep contract.

### §5.2 Recommendation for v2.2 (conditional)

**This recommendation is conditional on a structural close-the-1.18× gap being a hard v2.2 requirement.** If R3.3+ sub-2 stays parked (per sub-2B verdict §6.4: "Option C first, failing that re-decompose"), no backend question needs answering and this RFC is reference-only.

**Step 1 — Option A (Cranelift upgrade probe, 0.5–2 dw)**. Cheap; even if NEGATIVE on Gap #1, the upgrade is generally healthy and the test-build effort pays back via other Cranelift improvements (alias-analysis load-forward, aarch64 instruction selection). Frame as "1 dw probe, hard revert gate on regression". This is the SPG-methodology "do the cheap experiment first" pattern.

**Step 2 — Option D (Cranelift custom ISLE pass, 6–12 dw)** if Step 1 NEGATIVE on Gap #1. The custom-pass route closes the SPECIFIC named gap at lowest total effort. Risk is fork-maintenance overhead and CI complexity, but upstreaming the rule to bytecodealliance is a viable longer-term plan (3-month wall-time review cycle, not blocking). Gap #2 (`GlobalValueData` runtime-SSA) needs more work in the same fork but is enumerable.

**Step 3 — Re-decompose at a different layer (NOT a backend switch)** if Step 1 + 2 both NEGATIVE.

Per `v2.0-track-r-r3-3-design-space.md`, Design 2 (per-callsite immediate base shift) and Design 3 (deopt-only memory writes, never on the hot path) are layers that don't depend on mid-end store-DCE at all. If Cranelift's mid-end cannot be made to DCE the dual-write even with a custom pass, the right move is to NOT have a dual-write in the first place — change the IR shape so the same correctness is achieved without the redundant store. This is the perf-methodology §3 Round 4+ "re-decompose, don't polish the same attack surface" rule.

**Options B (LLVM JIT) and C (hand-asm) are NOT recommended for v2.2**:

- **B (LLVM)**: 12–26 dw timeline is multi-month senior eng time; 10–20× binary-size growth; CI matrix complexity. The wins it delivers (Gap #1 + Gap #2 + Gap #3 all closed via tbaa + noalias + DSE) ARE genuine and would unlock R3.3+ sub-2 path, but the cost is disproportionate to the gap (which is ~5 µs/cell on token_bucket per Phase F prediction). LLVM is justified ONLY if luna takes on additional multi-thread / advanced-optimization ambitions that Cranelift HEAD cannot service over 12+ months.
- **C (hand-asm)**: 16–32 dw per-arch + per-op. Solves Gap #3 trivially (no IR, no dual-write window) but gives up cross-arch portability and whole-trace optimizations. Only justified if luna decides Cranelift IR is fundamentally wrong for a trace JIT (a much larger redesign question than R3.3+ sub-2 closure). LuaJIT chose this path because it predates Cranelift's existence; greenfield projects in 2026 default to IR-based JIT.

### §5.3 Fallback path

If A NEGATIVE → try D. If D blocked (e.g. supply-chain / fork-maintenance constraints win out), the fallback is **Design 2/3 re-decomposition** under the existing Cranelift 0.124 backend. The R3.3+ sub-2 "dual-write" approach was Phase A's chosen design; if it's structurally incompatible with Cranelift's mid-end, change the design, not the backend.

This is the most honest answer the audit can give: **the gap is small enough (~5 µs/cell on token_bucket) that switching backends is overkill; the gap is large enough (Gap #1 + #2 falsified by two NEGATIVE attempts) that polishing the same shape is wasteful**. The right rung in the methodology ladder is to re-decompose at a different IR shape that doesn't depend on the specific Cranelift store-DCE behaviour.

### §5.4 luna-core 0-dep contract

ALL FIVE options preserve luna-core's 0-third-party-dep contract. Cranelift / LLVM / dynasm / cranelift-wasm deps would all live in luna-jit (and possibly luna-aot / luna-runtime-helpers). This is not a differentiator between options at the contract level; binary-size and CI complexity are the differentiators.

### §5.5 Honest closure

This RFC is a READ-ONLY OPTIONS AUDIT. It does NOT propose immediate backend switch. The matrix is the deliverable; the recommendation is conditional on the structural close-the-gap question being re-opened in v2.2. The recommended path (A → D → re-decompose) explicitly avoids the multi-month commitments (B, C) that would dominate v2.2 budget for a gap whose absolute size is ~5 µs/cell on token_bucket.
