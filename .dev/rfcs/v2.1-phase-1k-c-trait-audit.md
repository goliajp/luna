## § 1 现有 JitBackend trait 全清单 (luna-core 侧)

`crates/luna-core/src/jit/abi.rs:96-192` + `crates/luna-core/src/jit/storage.rs:35-62` + `crates/luna-core/src/jit/mod.rs:75-174`.

### § 1.1 `IntChunkCompiler` (method JIT) — `abi.rs:96-121`

```rust
pub trait IntChunkCompiler {
    fn try_compile(
        &self,
        storage: &mut dyn crate::jit::JitStorage,
        proto: Gc<Proto>,
        pre53: bool,
        float_only: bool,
    ) -> CompileResult;

    fn enter(&self, vm: *mut crate::vm::Vm, cl: Option<Gc<LuaClosure>>) -> JitVmGuard;

```

- **2 methods**, both ABI-pure (only luna-core types touched: `Gc<Proto>`, `CompileResult`, `JitVmGuard`, `Vm`, `LuaClosure`, the opaque `dyn JitStorage`).
- `CompileResult::Compiled` (`abi.rs:56-82`) carries 7 ABI-stable fields: `entry: *const u8`, `num_args: u8`, `returns_one: bool`, `arg_float_mask: u8`, `arg_table_mask: u8`, `ret_is_float: bool`, `ret_is_table: bool`.
- Entry pointer ABI shape locked by `IntChunkFn` / `IntFn1..4` aliases at `abi.rs:30-44`: `unsafe extern "C" fn(i64*N) -> i64`, with `MAX_JIT_ARITY = 4`.

### § 1.2 `TraceCompiler` (trace JIT) — `abi.rs:128-145`

```rust
pub trait TraceCompiler {
    fn try_compile_trace(
        &self,
        storage: &mut dyn crate::jit::JitStorage,
        record: &TraceRecord,
        opts: CompileOptions,

    fn last_compile_checkpoint(&self) -> &'static str;
```

- **2 methods**, both ABI-pure.
- `CompiledTrace` (`crates/luna-core/src/jit/trace_types.rs:549-587`) carries entry as `pub entry: TraceFn` where `pub type TraceFn = unsafe extern "C" fn(*mut i64) -> i64` (`trace_types.rs:470`). Plus `head_pc: u32`, `n_ops: u32`, `dispatchable: bool`, `window_size: u32`, `exit_tags: Vec<ExitTag>`.
- `last_compile_checkpoint()` is diagnostic-only — returns a `&'static str` bucket name. **No backend constraint** (LLVM impl picks its own checkpoint names, never compared to Cranelift's).

### § 1.3 `JitStorage` (per-Vm code-page owner) — `storage.rs:35-44`

```rust
pub trait JitStorage: std::any::Any {
    fn as_any_mut(&mut self) -> &mut dyn std::any::Any;
    fn as_any(&self) -> &dyn std::any::Any;
}
```

- **2 methods**, both downcast hooks.
- Designed exactly for this use case: the concrete impl (today: `CraneliftJitStorage`, tomorrow also: `LlvmJitStorage`) owns whatever code-holding structs the backend needs. luna-core never inspects the concrete shape; the JIT-side impl downcasts via `as_any_mut().downcast_mut::<MyStorage>()` and gets typed access to its own cache + handle Vecs.

### § 1.4 `JitVmGuard` (per-dispatch RAII) — `mod.rs:115-174`

Not a trait — a public struct with a `pub(crate) restore: Option<JitVmRebindRestore>` field. The restore record holds a `restore_fn: unsafe fn(prev_vm, prev_cl)` function pointer that the JIT side writes back on drop. luna-core constructs an inert (no-op) guard via `noop_jit_guard()`; backends construct a real one via `JitVmGuard::from_restore(JitVmRebindRestore { prev_vm, prev_cl, restore_fn })`.

**Critical**: this is the seam through which a backend installs *its own* TLS rebind logic without luna-core knowing about TLS. `JitVmRebindRestore::restore_fn` is a raw `unsafe fn` pointer; an LLVM backend can supply a different restorer that writes to a different TLS cell (or to no cell at all — the JIT helpers' TLS is the Cranelift backend's choice).

### § 1.5 Vm install API — `vm/exec.rs:1029-1058`

```rust

where S: JitStorage + 'static;
```

- **Already backend-generic.** Phase 1K.C does **NOT** widen this signature — an `LlvmBackend` slots in by-name into the same setters.

### § 2.1 `CraneliftBackend` struct + impls — `mod.rs:7285-7355`

```rust
#[derive(Clone, Copy, Debug, Default)]
pub struct CraneliftBackend;

impl IntChunkCompiler for CraneliftBackend {
    fn try_compile(&self, storage, proto, pre53, float_only) -> CompileResult {
        match cache_lookup_or_compile(storage, proto, pre53, float_only) {
            Some(tuple) => CompileResult::Compiled { ...unpack tuple... },
            None        => CompileResult::Skipped,
        }
    }
    fn enter(&self, vm, cl) -> JitVmGuard {

    }
}

    }
    fn last_compile_checkpoint(&self) -> &'static str {
        trace::last_compile_checkpoint()
    }
}
```

**The struct is a ZST**. Stateless. All state lives in `CraneliftJitStorage` (per-Vm) + the `JIT_VM` / `JIT_CL` thread-locals (per-thread).

### § 2.2 Compile pipeline — three layers

| Layer | Fn | File:line | Cranelift API used |
|---|---|---|---|
| **L1 — cache + handle parking** | `cache_lookup_or_compile` | `mod.rs:156` | `JITModule` (via storage downcast) |
| **L2 — JIT wrapper** | `try_compile_int_chunk` | `mod.rs:1684` | `JITBuilder::with_isa`, `JITModule::new`, `module.finalize_definitions()`, `module.get_finalized_function(fn_id)` |
| **L3 — backend-agnostic lowerer** | `lower_int_chunk_into<M: Module>` | `mod.rs:1746` | `cranelift_module::Module` trait + `cranelift::prelude::*` (`FunctionBuilder`, `InstBuilder`, `Value`, `Variable`, `types::I64`, `MemFlags`, …) |

- L2 = `try_compile_trace_with_options` at `trace.rs:3378`
- L3 = `lower_trace_into<M: Module>` / `lower_trace_into_named<M: Module>` at `trace.rs:3427`, `:3448`

### § 2.3 `M: cranelift_module::Module` — what does the generic abstract over?

`cranelift_module::Module` is **Cranelift's own** backend abstraction across `JITModule` (runtime JIT) and `ObjectModule` (luna-aot's static codegen). It defines:

- `declare_function(name, linkage, sig) -> FuncId`
- `declare_data(name, linkage, writable, tls) -> DataId`
- `define_function(fn_id, ctx) -> ModuleResult<()>`
- `define_data(data_id, desc) -> ModuleResult<()>`
- `make_context() -> Context`
- `target_config() -> TargetFrontendConfig`
- (plus 10+ more, see cranelift-module 0.124 docs)

**This trait is Cranelift-internal.** LLVM does not implement it and cannot — Cranelift IR (`cranelift_codegen::ir::Function`) is a different IR from LLVM IR (`inkwell::values::FunctionValue`). The `lower_int_chunk_into` / `lower_trace_into` generic bodies emit **Cranelift IR** through `FunctionBuilder::ins()`; they are not re-targetable to LLVM.

**Implication**: the L3 lowerers are not shared between Cranelift and LLVM backends. The LLVM backend writes its own L3 lowerer (over `inkwell::builder::Builder`); the Cranelift L3 lowerer stays unchanged and feeds both `JITModule` (luna-jit runtime) and `ObjectModule` (luna-aot static).

### § 2.5 Helper symbol registration — `mod.rs:1630`, `trace.rs:3222`

`build_jit_module_with_helpers()` (method JIT) registers 8 `luna_jit_*` symbols on a `JITBuilder` via `builder.symbol("name", addr as *const u8)`. `build_trace_jit_module()` (trace JIT) registers ~25 symbols (sets overlap; trace path needs `_table_set_field`, `_op_get_tab_up`, `_trace_materialize_frames`, `_op_concat`, `_str_buf_*`, etc.).

The `luna_jit_*` helpers themselves (`mod.rs:378..` — `luna_jit_new_table`, `luna_jit_table_set_int`, `luna_jit_upval_get`, `luna_jit_op_close`, `luna_jit_op_tforcall`, `luna_jit_stack_load`, `luna_jit_str_buf_intern`, …) are plain `#[unsafe(no_mangle)] pub unsafe extern "C" fn` functions. ### § 2.6 TLS rebind — `scoped_rebind.rs` (95 lines)

`scoped_jit_vm_rebind(vm, cl) -> JitVmGuard` captures the prior `(JIT_VM.get(), JIT_CL.get())`, installs the new values, and packages them into a `JitVmRebindRestore` whose `restore_fn` writes back on drop. The two TLS cells live in `mod.rs:288-296` (Cranelift backend's choice; not a luna-core concern).

### § 2.7 `SendJitModule` newtype — `send_jit_module.rs` (116 lines)

`#[repr(Rust)]` wrapper around `cranelift_jit::JITModule` with `Deref<Target = JITModule>` + `DerefMut` + `unsafe impl Send`. Justified because luna only constructs `JITModule` with the default `SystemMemoryProvider`, which is itself `Send`. This newtype is the J-A/J-E `Send` seam — it does not enter luna-core (lives entirely in luna-jit).

## § 3 LLVM 等价接入设计

### § 3.1 Crate shape — new sibling, not a feature inside luna-jit

**Decision**: introduce `crates/luna-jit-llvm` as a new sibling crate of `luna-jit`. NOT a `[features] llvm-jit = ["dep:inkwell"]` inside luna-jit.

Reasoning:
1. luna-jit ships with Cranelift unconditionally — embedders relying on the existing `cargo install luna-jit` get the same crate (size, dep tree, binary surface). A feature flag inside luna-jit would silently change build cost depending on a flag's presence; sibling crate makes the LLVM opt-in explicit at the dep level.
2. Phase 1K.B verdict § 6 already recommended sibling-crate path. Reaffirmed.
3. `cargo publish` story: luna-jit 1.x → crates.io stays cranelift-only. luna-jit-llvm 1.x ships separately; embedders pull both if they want both.

Crate layout:

```
crates/luna-jit-llvm/
  Cargo.toml          # depends on luna-core (path/version) + inkwell 0.9 (llvm18-1)
  src/
    lib.rs            # public surface: LlvmBackend, install_llvm_jit, new_minimal_with_llvm_jit
    jit_backend/
      mod.rs          # LlvmBackend struct + IntChunkCompiler impl + helper symbol registration
      trace.rs        # TraceCompiler impl + trace lowerer
      storage.rs      # LlvmJitStorage (impl JitStorage)
      scoped_rebind.rs # llvm-side TLS rebind (mirrors cranelift's)
```

### § 3.2 `LlvmBackend` struct + impls (target shape)

```rust
// crates/luna-jit-llvm/src/jit_backend/mod.rs
use inkwell::context::Context;
use luna_core::jit::{IntChunkCompiler, TraceCompiler, CompileResult, JitVmGuard, ...};

#[derive(Clone, Copy, Debug, Default)]
pub struct LlvmBackend;

impl IntChunkCompiler for LlvmBackend {
    fn try_compile(
        &self,
        storage: &mut dyn luna_core::jit::JitStorage,
        proto: Gc<Proto>,
        pre53: bool,
        float_only: bool,
    ) -> CompileResult {
        match llvm_cache_lookup_or_compile(storage, proto, pre53, float_only) {
            Some((entry, num_args, returns_one, fmask, tmask, ret_f, ret_t)) =>
                CompileResult::Compiled { entry, num_args, returns_one,
                                          arg_float_mask: fmask, arg_table_mask: tmask,
                                          ret_is_float: ret_f, ret_is_table: ret_t },
            None => CompileResult::Skipped,
        }
    }
    fn enter(&self, vm: *mut Vm, cl: Option<Gc<LuaClosure>>) -> JitVmGuard {
        let vm_ref = unsafe { &mut *vm };
        scoped_rebind::scoped_llvm_jit_vm_rebind(vm_ref, cl)
    }
}

impl TraceCompiler for LlvmBackend {
    fn try_compile_trace(&self, storage, record, opts) -> Option<CompiledTrace> {
        trace::try_compile_trace_with_options_llvm(storage, record, opts)
    }
    fn last_compile_checkpoint(&self) -> &'static str {
        trace::last_compile_checkpoint_llvm()
    }
}
```

**Zero changes** required to `IntChunkCompiler` / `TraceCompiler` traits. The two impls slot in via the existing setter `vm.install_jit_backend(LlvmBackend, LlvmBackend)`.

### § 3.3 `LlvmJitStorage` — what does it own?

```rust
// crates/luna-jit-llvm/src/jit_backend/storage.rs

pub(crate) struct LlvmJitStorage {
    // Per-Vm LLVM context. !Send by default; the storage's
    // !Send-ness is fine because Vm itself is !Send pre-J-D-flip.
    // Held inside an Option so we can move it out for compile calls
    // (inkwell APIs borrow Context immutably for Module creation).
    pub(crate) ctx: Context,

    // Per-Vm ExecutionEngine pool — one EE per compiled Module
    // because inkwell 0.9's `create_jit_execution_engine` consumes
    // the Module by-value into the EE. Each entry holds the mmap'd
    // mcode for one chunk-or-trace compile.
    //
    // Alternative considered: single shared ORC v2 `LLJit` instance.
    // See § 3.4 for the JIT engine model decision.
    pub(crate) compiled_units: Vec<LlvmCompiledUnit>,

    // Same bytecode-key → CompileResult cache as Cranelift backend.
    // Key + entry shape are identical; only the owner of the mcode
    // mmap differs (LlvmCompiledUnit vs Cranelift's JitHandle).
    pub(crate) cache: HashMap<u64, CacheEntry>,
}

pub(crate) struct LlvmCompiledUnit {
    // Owns the ExecutionEngine which owns the mcode mmap pages.
    // Drop releases the pages.
    _ee: inkwell::execution_engine::ExecutionEngine<'static>,
    _entry_raw: *const u8,
}

impl JitStorage for LlvmJitStorage {
    fn as_any_mut(&mut self) -> &mut dyn std::any::Any { self }
    fn as_any(&self) -> &dyn std::any::Any { self }
}
```

**Lifetime gap (TOP-3 risk #1)**: inkwell's `Context<'ctx>`, `Module<'ctx>`, `Builder<'ctx>`, `ExecutionEngine<'ctx>` are all parametrised over a single `'ctx` lifetime that borrows the `Context`. Storing them all in one struct requires either:
- (a) self-referential struct via `ouroboros` or hand-rolled lifetime extension — adds dep + soundness reasoning.
- (b) `unsafe { std::mem::transmute::<ExecutionEngine<'_>, ExecutionEngine<'static>>(ee) }` — pinned by the field-drop order in the struct (drop EE before Context). The probe crate doesn't exhibit this because it creates Context fresh per `jit_add` call.
- (c) One `Context` per compile call, throw away after JIT-finalize. Wastes the LLVMContext init cost (~1-5 ms per `LLVMContextCreate` from `llvm-sys` calls) per Proto compile, but trivially correct.

**Recommended**: (b) with field-drop-order documented + a `#[must_use]` constructor that enforces it. The transmute is local; the `Context` outlives the `ExecutionEngine` by construction (both owned by same `LlvmJitStorage`, dropped in declaration order — Rust drops fields top-to-bottom by default, so put `compiled_units` before `ctx` in the struct).

Confirm at 1K.D entry: read inkwell 0.9 `ExecutionEngine`'s `Drop` impl + `from_*` constructors to verify the `'ctx` borrow is purely advisory (the mmap is owned by LLVM's internal allocator, released on `LLVMDisposeExecutionEngine`, which doesn't touch the Context).

### § 3.4 JIT engine model — per-Module EE vs shared LLJit ORC v2

**Option A — per-Module EE** (what the probe does): each compile creates a fresh `Context::create_module(...)`, builds IR, calls `module.create_jit_execution_engine(opt)`. The EE owns the mmap; one EE per compiled fn. Method JIT cache holds one EE per Proto.

- Pros: matches probe pattern, simplest mapping, no ORC v2 concepts to learn.
- Cons: EE construction has fixed overhead per compile (~hundreds of µs to set up `MCJIT` per the LLVM docs); luna's method-JIT compile rate is ~100s of Protos/Vm so the overhead is bounded.

**Option B — shared LLJit ORC v2** (`inkwell::orc2::LLJit`): one long-lived JIT engine per Vm. Each compile adds a `ThreadSafeModule` into the JIT; symbols resolve lazily through the JIT's dylib search order.

- Pros: lower per-compile overhead, better symbol management (single shared resolver for all `luna_jit_*` helpers).
- Cons: inkwell 0.9's ORC v2 wrapper coverage is partial (per § 1K.A § 2.2 row "ORC v2 / LLJit 支持: YES"); some lower-level ORC features may still require dropping to `llvm-sys` for fine-grained control.

**Recommended**: **Option A** for 1K.D framework impl (mirror probe pattern, lowest impl risk). Migrate to Option B in 1K.G if per-compile overhead measurably eats into the predicted ~10–30 µs/cell gain (per § 1K.A § 1.2 ceiling). 1K.G migration is cheap because the trait-impl seam at `LlvmBackend` doesn't change — only `LlvmJitStorage`'s internals.

### § 3.5 Helper symbol registration

The `luna_jit_*` symbols live in `luna-jit` (`jit_backend/mod.rs:378..`). luna-jit-llvm cannot easily reference them across crate boundaries because:
- They are `#[unsafe(no_mangle)] pub unsafe extern "C" fn`. The `no_mangle` makes them globally addressable by name at runtime via `dlsym(RTLD_DEFAULT, "luna_jit_new_table")`.
- However, if `luna-jit` is linked as an rlib (not cdylib) into the final binary, the linker strips unused `no_mangle` symbols. The Cranelift backend works around this by explicitly registering each symbol via `JITBuilder::symbol("name", addr as *const u8)` — see `mod.rs:1646-1668`.

**LLVM equivalent**: `ExecutionEngine::add_global_mapping(&fn_value, addr)`. Each `luna_jit_*` helper must be (a) `declare`d in the LLVM module with the right signature, then (b) mapped to the runtime address via `add_global_mapping`.

**Cross-crate access pattern**: luna-jit-llvm needs to depend on luna-jit *with the helper symbol fn pointers re-exported* — OR — luna-jit-llvm depends on luna-core only (no luna-jit), and the `luna_jit_*` helpers are moved/copied into a third "luna-jit-helpers" crate that both backend crates depend on.

**Recommended path** (cleanest, 1K.D step): introduce `crates/luna-jit-helpers` as a small crate containing only the 26+ `luna_jit_*` `extern "C"` fns + the `JIT_VM` / `JIT_CL` thread locals + `scoped_jit_vm_rebind`. Both `luna-jit` (Cranelift) and `luna-jit-llvm` re-export and register from there. luna-core 0-dep contract unaffected (helpers crate depends on luna-core, not the other way around).

**Alternative** (lower-touch but uglier): luna-jit-llvm directly `extern "C" { fn luna_jit_new_table() -> i64; ... }` declarations + link `cargo:rustc-link-lib=static=luna_jit` via build.rs. This works but couples link order to discovery and breaks `cargo install luna-jit-llvm` for users who don't link luna-jit as a staticlib.

1K.D-A entry: spend ~half a dev-day extracting `crates/luna-jit-helpers`. luna-jit still re-exports the helper fns from it (`pub use luna_jit_helpers::*`) so existing call sites stay green. This is the **first 1K.D commit** before any LLVM code lands.

### § 3.6 `JitVmGuard` reuse

LLVM backend supplies its own `restore_fn: unsafe fn(prev_vm, prev_cl)` that writes the captured pair back into the shared TLS cells. If luna-jit-helpers (per § 3.5) owns the TLS cells + the rebind function, **both backends literally share the same `scoped_jit_vm_rebind` impl** — `LlvmBackend::enter` becomes:

```rust
fn enter(&self, vm: *mut Vm, cl: Option<Gc<LuaClosure>>) -> JitVmGuard {
    let vm_ref = unsafe { &mut *vm };
    luna_jit_helpers::scoped_jit_vm_rebind(vm_ref, cl)
}
```

— identical to `CraneliftBackend::enter`'s body. No backend-specific TLS at all.

### § 3.7 IR lowerer (the actual codegen work)

This is **the bulk of 1K.D through 1K.G effort** and is **NOT** designed in 1K.C (scope discipline). 1K.C only confirms the trait surface accommodates it.

Sketch only: the LLVM int-chunk lowerer's body looks like the probe's `jit_add` blown up to handle the cumulative whitelist (~20 ops: `Move`, `LoadI`, `LoadF`, `Add`, `Sub`, `Mul`, `Div`, `Mod`, `Lt`, `Le`, `Eq`, `Jmp`, `ForPrep`, `ForLoop`, `Return0/1`, `Call` self-recursive, `GetUpval`, `NewTable`, `SetI`, `GetI`, `Len`). Cranelift's lowerer at `lower_int_chunk_into` is ~3300 LOC; the LLVM equivalent at the same op coverage is estimated ~3500-4500 LOC in Rust (inkwell builder calls are slightly more verbose than `bcx.ins().*` due to Result-returning fallible builders).

Trace lowerer (`lower_trace_into`, ~6500 LOC at `trace.rs:3427..9000`) is the bigger lift. 1K.E (or earlier sub-step) starts with a tiny single-op trace lowerer; reach full op coverage incrementally.

### § 3.8 Trait extension assessment — **NO** extension needed

The audit asks whether `IntChunkCompiler` / `TraceCompiler` / `JitStorage` need new methods to accommodate LLVM. Verdict:

- **`IntChunkCompiler` / `TraceCompiler`**: NO. All inputs (`Proto`, `TraceRecord`, `CompileOptions`) and outputs (`CompileResult`, `Option<CompiledTrace>`) are ABI-pure. LLVM impl reads/writes the same shapes.
- **`JitStorage`**: NO. The `Any` downcast pattern is exactly the extensibility seam for new storage types.
- **`JitVmGuard`**: NO. The `JitVmRebindRestore { restore_fn }` function-pointer seam already lets backends supply their own TLS restorer (though in practice both backends reuse the same one via `luna-jit-helpers`).

**Optional, diagnostic-only nice-to-have (NOT required for ship)**: a `fn backend_kind(&self) -> &'static str` default method on `IntChunkCompiler` returning `"cranelift"` / `"llvm"` for log lines. Pure-additive, default impl returns `"unknown"`, zero ABI risk. Defer to 1K.H when the dual-backend bench harness needs to print which backend produced a number. **Not scheduled for 1K.D.**

## § 4 双 backend 切换机制 — Cargo feature + env var (hybrid)

### § 4.1 Three candidate designs

| Design | Selection point | Both backends in binary? | Default-build LLVM cost? |
|---|---|---|---|
| **D1 — env var only** `LUNA_JIT_BACKEND=cranelift\|llvm` | Runtime | Yes, always | Yes (every binary carries LLVM 18) |
| **D2 — cargo feature only** `--features llvm-jit` | Compile time | No (one-or-the-other) | No (default = cranelift only) |
| **D3 — hybrid: feature gates inclusion + env var picks within feature** | Compile time + runtime | When feature on | No (default = cranelift only) |

### § 4.2 Recommended = D3 hybrid

Reasoning:

1. **luna-core 0-third-party-dep contract** + **luna-jit cranelift-only default** must not regress. D1 forces LLVM 18 into every `cargo install luna-jit` user's binary — unacceptable.
2. D2 alone prevents A/B comparison within a single test run (would need two cargo builds + two test invocations). The Phase 1K.H bench plan compares cranelift vs llvm head-to-head on token_bucket_1k; running both in one process with `LUNA_JIT_BACKEND=...` flips is ergonomic.
3. D3 keeps default builds cranelift-only (no LLVM dep) but enables runtime A/B when the embedder opts in at compile time. Best of both.

### § 4.3 D3 implementation sketch (NOT implemented in 1K.C)

```toml
# crates/luna-jit/Cargo.toml
[features]
default = []
llvm-jit = ["dep:luna-jit-llvm"]   # pulls the sibling crate

[dependencies]
luna-jit-llvm = { path = "../luna-jit-llvm", version = "= 1.3.0", optional = true }
```

```rust
// crates/luna-jit/src/lib.rs
pub fn install_default_jit(vm: &mut vm::Vm) {
    #[cfg(feature = "llvm-jit")]
    {
        if std::env::var_os("LUNA_JIT_BACKEND")
            .map(|v| v == "llvm")
            .unwrap_or(false)
        {
            luna_jit_llvm::install_llvm_jit(vm);
            return;
        }
    }
    // Default = Cranelift, identical to today's behaviour.
    vm.install_jit_backend(jit_backend::CraneliftBackend, jit_backend::CraneliftBackend);
    vm.install_jit_storage(jit_backend::storage::CraneliftJitStorage::default());
}
```

CI matrix:
- existing default job — cranelift only (unchanged).
- new `llvm-jit` job — `cargo test --features llvm-jit` runs both backends; per-test selection via `LUNA_JIT_BACKEND` env.
- new bench job — both backends side-by-side on token_bucket_1k (Phase 1K.H scope).

### § 4.4 What about `runtime switching` (a.k.a. swap one backend for another mid-Vm)?

Out of scope. Per the existing `install_jit_storage` docstring (`exec.rs:1049`): "re-installing storage on a Vm that already holds compiled-trace pointers WILL evict their owners." Switching backends mid-Vm means dropping all compiled mcode — only useful for tests, not production. 1K.D-1I do NOT need it. Mention in 1K.H test plan only as a "don't bother".

## § 6 Top-3 风险

### Risk 1 — inkwell 0.9 `'ctx` lifetime + per-Vm Storage self-reference

**Severity**: HIGH. Materialises at 1K.D.7 (storage + cache).

**Description**: `Context<'ctx>` borrows from itself; `Module<'ctx>`, `Builder<'ctx>`, `ExecutionEngine<'ctx>` all carry that lifetime. luna's per-Vm storage needs to own the Context AND multiple ExecutionEngines (one per compiled Proto). Rust's borrow checker rejects this trivially without:
- `ouroboros` self-referential macro (new dep, requires unsafe internally), OR
- hand-rolled `transmute<ExecutionEngine<'_>, ExecutionEngine<'static>>` with strict field-drop order discipline (per § 3.3 option (b)), OR
- one-Context-per-compile throwaway pattern (per-compile Context init cost).

**Mitigation**: pre-1K.D micro-spike. Spend ~half-day at start of 1K.D.7 writing a 50-line standalone test that holds a `Context` + a `Vec<ExecutionEngine<'static>>` in one struct with the transmute trick, asserts mcode survives across compiles, runs Miri or at minimum dropck-stress. If transmute path fails Miri, fall back to per-compile Context (slower but trivially correct).

**Fallback**: if the spike rules out both transmute and per-compile patterns, escalate scope: drop to `llvm-sys` direct (per 1K.A § 2.3 备选 trigger A) where raw `LLVMContextRef` ptrs are easy to own. Adds ~1 dev-week to 1K.D-1I.

### Risk 2 — LLVM trace deopt / side-exit SSA shape mismatch with Cranelift

**Severity**: HIGH. Materialises at 1K.G.

**Description**: Cranelift's `lower_trace_into` lays out side-exit blocks (cmp-guard failure → write `reg_state` → return early-PC) with `brif` + side-block emit + early `return`. LLVM IR's structured-block + dominance requirements + `LandingPad` semantics may force a different control-flow shape (e.g. each side-exit becomes its own basic block reachable only through a `br` from the guard; phi nodes at re-join sites become explicit `alloca`+store / `load`+phi). The trace lowerer is luna's largest single codegen file (~6500 LOC) and the side-exit machinery is its most subtle part.

**Mitigation**: at 1K.G entry, design the LLVM side-exit shape on paper first (one-page sketch, not code) covering: cmp guard, reg_state-store-then-return, exit_tags propagation, sunk-table materialise helper-call shape. Validate against a single failing-guard trace in 1K.G.1 before scaling. If the SSA shape can't be made to honour the existing `TraceFn = unsafe extern "C" fn(*mut i64) -> i64` ABI (it should — the ABI is solid), escalate to trait extension (a new `LlvmTraceFnVariant` ABI shape) — but that breaks luna-core's "no backend-specific ABI" cleanliness; treat as last resort.

**Fallback**: if 1K.G overruns by >2 dev-weeks, ship Phase 1K with int-chunk LLVM only (skip trace JIT) and defer trace-LLVM to a v2.2 follow-on RFC. Method-JIT-only LLVM still demonstrates the trait surface works for both backends; trace JIT closure via LLVM becomes a future hardening.

### Risk 3 — `luna-jit-helpers` extraction breakage radius

**Severity**: MEDIUM. Materialises at 1K.D.1.

**Description**: Extracting 26+ `luna_jit_*` extern "C" fns + 2 TLS cells + `scoped_jit_vm_rebind` from `luna-jit/src/jit_backend/mod.rs` (where they currently live, ~5800 LOC in that file out of the 7355) into a new sibling crate touches many internal call sites. Each helper is itself ~10-50 LOC and references `luna_core::*` types — the new crate's `Cargo.toml` needs to depend on luna-core. luna-jit then `pub use luna_jit_helpers::*` to preserve all existing call paths. Risk = subtle breakage of `JITBuilder::symbol("name", helper as *const u8)` registration (the `helper as *const u8` cast pulls the fn from the wrong crate's symbol table if visibility changes).

**Mitigation**: structure D.1 commit so all 377 workspace tests run green on the same commit that introduces the new crate. Do NOT split D.1 into "create crate" + "migrate fns" — atomicity matters because half-migrated state is hard to test. The `pub use luna_jit_helpers::*` re-export in luna-jit's `jit_backend/mod.rs` keeps every existing `crate::jit_backend::luna_jit_new_table` path resolving unchanged.

**Fallback**: if extraction triggers >5 tests' worth of fallout that can't be cleared in ~1 dev-day, abandon the helpers-crate path and fall back to § 3.5 alternative ("luna-jit-llvm extern "C" declarations + static link"). Higher coupling but less surface to bend.
