### § 1.1 The residual ~16 µs/cell unattributed cost

Phase 1I.D § 3.1 enumerates three sub-hypotheses for the residual:

1. **I-cache locality** — inline IC body (52 insns / iter per the
   asm decomp) co-resident in the trace's I-cache footprint vs the
   warmed helper body that lives in a separate L1i line.
2. **LSU pipeline interaction** — IC's dependency chain (load
   nodes_ptr → load key from slot → compare to baked key const →
   load val_raw) traverses 3 dependent loads, all in the LSU's
   issue window for the 1k-iter steady state.
3. **Key-const materialization** — IC bakes the `Gc<LuaStr>` key
   pointer via 3-µop `movz` + 2× `movk` on aarch64 (per `asm-decomp.md`
   block3); the helper resolves the same pointer via a single
   PC-relative literal load (`ldr xN, =literal`), saving 2 µops.

Without a PMU probe, all three are plausible; their relative
weights are unknown. Phase 1J's candidate audits treat them as
equally weighted and note candidate-specific applicability per item.

## § 3 Phase 1J.3 — LLVM JIT codegen quality audit

### § 3.1 Does LLVM close the unattributed ~16 µs/cell?

Per `v2.1-cranelift-alternative-backend-rfc.md` § 3.1, LLVM JIT (via
inkwell) closes:

- **R3.3+ Gap #1**: same-address back-to-back store DCE — addresses
  the dual-write window in trace JIT lowering, not directly the IC
  residual.
- **R3.3+ Gap #2**: `noalias` + `!alias.scope` metadata replaces
  `MemFlags::trusted()` — addresses the alias-region restriction
  that drove sub-2B revert, not directly the IC residual.
- **R3.3+ Gap #3**: LLVM DSE eats dual-write when alias info
  confirms — addresses the dual-write cost, not directly the IC
  residual.

For the Phase 1I.D unattributed ~16 µs/cell specifically, LLVM
closes the components as follows:

| Sub-hypothesis | LLVM closure mechanism | Plausible gain |
|---|---|---|
| I-cache locality | LLVM's post-RA instruction scheduler aggressively interleaves to reduce I-cache pressure; LLVM also supports `-Os` style profile-guided layout. | ~3–5 µs |
| LSU pipeline interaction | LLVM's SchedModel for Apple Silicon (in upstream LLVM Apple M-series target) hand-tunes load latency; cranelift's aarch64 schedmodel is minimal. | ~5–8 µs |
| Key-const materialization | LLVM emits ARM64 `MOVZ + MOVK` chains identically to cranelift for the `Gc<LuaStr>` baked-const pattern; the PC-relative literal path requires constpool emit which LLVM does default-on for 64-bit constants. | ~2–3 µs |

Plausible aggregate: **~10–16 µs/cell on the IC residual**, plus the
~5–7 µs/cell from closing the R3.3+ Gap #1/#2/#3 components on
non-IC trace shapes. Total: ~15–23 µs/cell on token_bucket_1k under
the inline-IC code path.

### § 3.3 Family-1 re-assessment

Charter §188 frames LLVM as RESOLVING Family-1. Re-reading the
Phase 4 closure (`v2.0-pi-phase4-verdict.md`) — Family-1 is
rustc-LLVM-at-O3 pessimize on `luna-core`'s `Vm::run` op dispatch
hot loop. Switching the **JIT backend** to inkwell-LLJIT changes
which optimizer compiles the **trace JIT'd code**; it does NOT
change which optimizer compiles `Vm::run` (still rustc). So:

- Family-1 attacks that pessimized `Vm::run` (A8 = added arm; S07 = hash polish in interp; O04 = optimization in interp; A5-D1 = dispatch-table) DO NOT get unlocked by LLVM JIT backend.
- The "LLVM-as-JIT-backend resolves Family-1" framing in charter §188 is **misleading**. It conflates two LLVMs (the rustc-compile-time one and the inkwell-runtime one).
- Phase 1J.4's helper inlining attack is in luna-jit, not luna-core; Family-1 risk is genuinely LOW there because the pessimize surface is `Vm::run` op dispatch which is unchanged.

This re-assessment affects Phase 1J.6 ranking: LLVM JIT migration does NOT have the side benefit of unlocking the 5 NEGATIVE-reverted Phase 4 attacks, contrary to the charter claim. Its perf-attack ROI is bounded by the R3.3+ Gap closures + the ~16 µs/cell IC residual closures = ~20–30 µs/cell on token_bucket_1k. That gain budget on a 12–26 dw effort is **way below the methodology gate** (gain ≥ 5 µs/cell over ≤ 1 dw effort, scaling linearly upward).

### § 3.4 GO / NO-GO recommendation

**NO-GO for Phase 1J.B**. Effort 12–26 dw vs gain ceiling
~20–30 µs/cell on token_bucket is a 0.8–2.5 µs/cell-per-dw budget,
far below the Phase 1H methodology gate. Defer to v2.2 IF multi-thread
or advanced-opt becomes an independent requirement; the perf-attack
justification alone does not carry the cost.

## § 4 Phase 1J.4 — Helper inlining specialization audit

### § 4.1 The 8 helper sites + the inlining-vs-IC distinction

Per Phase 1H § 5.2, token_bucket_1k's hot loop body has 8 helper
calls per iter. Phase 1I tried to ELIMINATE these via inline IC
(materialise the cache-hit path in trace IR). That attack
NEGATIVE'd: 1-site IC -20 µs/cell vs env-OFF; multi-site IC
-170 µs/cell vs env-OFF (Phase 1I.C). The mechanism was Cranelift
regalloc2 spill on the IC's added Variables.

Helper inlining **specialization** is a different attack: keep the
helper call boundary (same regalloc behaviour as the existing
helper) but specialise the helper body for the trace recorder's
guarantees:

- The trace recorder already verifies `mt.is_none()` at trace
  entry. The helper's `g.metatable().is_some()` check is a redundant
  per-call test.
- The trace dispatcher enters via `enter_jit(vm, Some(cl))` which
  pins `JIT_VM` and `JIT_CL` for the dispatch window. The helper's
  `current_jit_vm()` TLS lookup repeats per call; could be hoisted
  to a per-trace-entry constant baked into IR.
- The helper's `pending_err.is_some()` check is needed only at
  trace-entry; the trace's guard chain already prevents pending_err
  from being set mid-trace (any sub-call that sets it deopts via
  sentinel). Could be moved to trace-entry.

### § 4.2 Specialised helper bodies — sketch

```rust
// Existing
pub unsafe extern "C" fn luna_jit_table_get_field(t: i64, key_ptr: i64) -> i64 {
    let vm = unsafe { current_jit_vm() };          // TLS lookup
    if vm.jit.pending_err.is_some() { return 0; }  // branch + load
    let g: Gc<Table> = Gc::from_ptr(t as *mut _);
    if g.metatable().is_some() {                   // branch + load
        vm.jit.pending_err = Some(vm.rt_err("..."));
        return 0;
    }
    let key_gc: Gc<LuaStr> = Gc::from_ptr(key_ptr as *mut _);
    let v = g.get(Value::Str(key_gc));
    let (_tag, raw) = v.unpack();
    unsafe { raw.zero as i64 }
}

// Specialised
pub unsafe extern "C" fn luna_jit_table_get_field_no_mt(
    t: i64, key_ptr: i64, /* vm and key both baked as IR consts */
) -> i64 {
    let g: Gc<Table> = Gc::from_ptr(t as *mut _);
    let key_gc: Gc<LuaStr> = Gc::from_ptr(key_ptr as *mut _);
    let v = g.get(Value::Str(key_gc));
    let (_tag, raw) = v.unpack();
    unsafe { raw.zero as i64 }
}
```

Cost saved per call: TLS lookup (~5–10 ns) + pending_err read +
branch (~3 ns) + metatable load + branch (~3–5 ns) ≈ **11–18 ns**.

Per-iter: 8 calls × ~14 ns = ~112 ns / iter ≈ **~112 µs/cell** on
token_bucket_1k.

Methodology atomic-op-table check (`~/.claude-shared/global/methodology/perf-decomposition-vs-polish.md` § 4):

| Op saved | Cost per call | Calls / iter | Saved µs / cell |
|---|---|---|---|
| TLS load (Apple Silicon vDSO-style mrs+add) | ~5 ns | 8 | ~40 |
| pending_err load + cmp + brif | ~3 ns | 8 | ~24 |
| metatable load + cmp + brif | ~5 ns | 8 | ~40 |
| **Subtotal** | | | **~104 µs/cell** |

The estimate is within ±20 % of the back-of-envelope ~112 µs/cell,
satisfying the methodology budget rule.

### § 4.3 Trace recorder guard prerequisites

For the specialised helper to be correct, the trace recorder MUST
emit an entry-guard that proves:

1. The target table's metatable is `None` at trace entry, AND
2. The table's metatable did not change during recording.

Phase 1I.B already implements both: per § 3.1 of the 1I.D doc, the
trace recorder snapshot verifies `mt.is_none()` for the IC site.
This same snapshot mechanism extends to a per-call guard before
each specialised helper invocation.

### § 4.4 Family-1 risk re-assessment

Per § 3.3, Family-1 = rustc-LLVM-at-O3 pessimize on luna-core's
`Vm::run` op dispatch. Helper inlining specialization is in
luna-jit (`jit_backend/mod.rs` + `jit_backend/trace.rs`) — does
**not touch `Vm::run`**, does not add new op-arms, does not change
the dispatch discriminant layout. Family-1 risk = **LOW**.

The one Family-1-adjacent risk: trace IR's emit site for each
helper call needs to add a guard to choose between baseline helper
vs specialised helper at compile time. If the guard logic adds
branches in trace.rs's `lower_trace_into_named` hot path, cranelift's
own regalloc may stress; but this is in the trace COMPILER, not the
trace RUNTIME, and compile-time overhead on token_bucket is not the
hot path (trace compile is one-shot per unique trace).

### § 4.7 Failure mode + revert plan

If 1J.B bench validates NEGATIVE, the failure modes most likely:

1. Specialised helpers consume more I-cache footprint than the
   single baseline helper (8 specialised + 8 baseline = 16 bodies
   resident), pushing key code out of L1i. Mitigation: have just
   one specialised helper variant per kind (get vs set), not per
   call site.
2. The entry-guard cost exceeds the per-call savings on cold
   workloads. Mitigation: gate via env knob (`LUNA_JIT_HELPER_SPEC=1`)
   default OFF, ship-on-validation.
3. The `Value::unpack` + `raw.zero as i64` path inside the helper
   becomes more visible at hot-path level once the surrounding
   cost is removed; the next decomposition would re-attack the
   `Value` layout, looping back to the BLOCKED candidate § 5.

Revert path: same as 1I.D — `git revert` the implementation commit;
ship docs as REVERT verdict. Per § 4.5, no luna-core change, so no
cross-crate impact on revert.

## § 5 Phase 1J.5 — Value layout flatten audit

### § 5.1 Charter §40 / §113 context

Per `v2.1-charter-118-current-audit.md` § "Smallest multi-month
path", Value-layout shrink (NaN-boxing) closes the `Value::raw_eq`
cluster (~14.7 µs/cell) + Table internals (~32 µs/cell where
internals cost is dominated by 16-byte Value comparisons inside
`main_position` + Robin Hood probe). Conservative projection:
closes ~25–35 µs/cell of the ~150 µs/cell bare-metal gap (~17–23 %
of the gap).

**BLOCKED** per ship-readiness :58: P02 16-byte invariant + 0-unsafe-emb
double red-line.

### § 5.2 Does it close the IC residual ~16 µs/cell?

Partially. Per Phase 1I.C § 3.1 #3, "if the Value enum's tag-and-raw
can be unioned into a single i64, reg pressure drops by ~50 % and
the IC body has room to breathe without spill." The 1I.D NEGATIVE
on 1-site IC was reg-pressure spill (~6 ns / iter STLF round-trip
on val_raw via stack slot, per § 2.3 of the 1I.D doc).

Sub-component closure:

| 1I.D sub-hypothesis | NaN-box closure | µs gain on the 16 |
|---|---|---|
| I-cache locality | indirect — fewer bytes per Value means tighter trace body | ~2 |
| LSU pipeline | direct — half the loads per slot probe | ~4–6 |
| Key-const materialization | unchanged (Gc<LuaStr> ptr is the cache key regardless of Value layout) | 0 |

Partial closure: ~6–8 µs/cell of the 16 µs IC residual, plus the
charter §113 cluster's ~25–35 µs/cell on interp dispatch (separate
attack surface).

### § 5.3 Partial flatten feasibility

Can luna's Value enum be flattened to a hot subset (Int / Str /
Bool) without breaking the 16-byte invariant?

Audit findings (read-only against `crates/luna-core/src/runtime/value.rs`
inferred from charter + ship-readiness refs; not directly Read in this
audit):

- The 16-byte invariant per P02 is for the `RawVal` payload — the
  Phase TB userdata trait + the trace JIT's reg_state Variable
  layout both assume `Value = (tag: u8, raw: 16 bytes)`.
- NaN-boxing reduces this to `Value = u64`. Float values get NaN-tagged;
  Int/Str/Bool/Nil/Userdata/Table/Function get encoded in non-NaN
  bit-patterns.
- Embedded API (`luna-core` public surface): per CLAUDE.md
  "0 unsafe at embedder surface", any user-API embedder cannot
  require `unsafe` block. NaN-boxing internally uses transmute /
  bit-cast but does NOT expose unsafe at the embedder API — the
  embedder still calls `Value::Int(42)` or `value.as_int()`. So
  0-unsafe-emb is preserved.
- The P02 16-byte invariant is the genuine blocker. Breaking it
  requires:
  - Re-shape trace JIT's reg_state Variable layout (8 bytes per
    slot instead of 16; reg pressure halves per § 5.2).
  - Re-shape `Vm::run`'s op dispatch — every site that reads
    `regs[A].tag` and `regs[A].raw` separately must learn the new
    encoding. This is the **Family-1 surface**.
  - Re-shape Table internals — `main_position` + Robin Hood probe
    operate on 16-byte Value compares; new 8-byte compares is a
    structural rewrite.
  - Re-shape `Gc<T>` interactions — currently the Gc raw pointer
    lives in `RawVal.gc` field; NaN-boxing must encode the Gc ptr
    in a sub-bit-pattern that doesn't collide with float NaN
    payload bits.

### § 5.4 Family-1 risk re-assessment

Per § 3.3, Family-1 = rustc-LLVM-at-O3 pessimize on `Vm::run`'s
op dispatch hot loop. Value-layout flatten **directly modifies the
dispatch discriminant load** at every op-arm. This is the EXACT
surface Phase 4's 5 NEGATIVE-reverted attacks (A8 / hash-polish /
S07 / O04 / A5-D1) pessimized. Family-1 risk = **HIGH**.

Mitigations:

1. Build the new dispatcher behind a feature flag, bench-validate
   on token_bucket + fib(28) + 9 RFC cells BEFORE committing.
2. Use Phase 6's tag-byte-unsigned-i64 IR Variable representation
   (per A5 Design 2/3 RFCs); this lets cranelift's mid-end CSE
   the discriminant load at trace JIT compile time, mitigating
   one of the Family-1 mechanisms.

But neither mitigation is a guarantee. Phase 4's verdict was that
ANY new dispatcher discriminant variation pessimizes rustc-LLVM's
codegen on `Vm::run`. Value-layout flatten is a larger variation
than the 5 NEGATIVE attacks were.
